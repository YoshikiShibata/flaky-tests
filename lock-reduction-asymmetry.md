# 排他制御の削除・縮小リファクタの非対称リスク

## 概要

並列実行される処理（テスト、ジョブ、RPC ハンドラ、バックグラウンドワーカー等）の性能改善目的で「不要に見える排他制御を外す」リファクタを行った際、本当は必要だった保護を巻き込みで削除してしまい、潜在バグや Flaky を作り込んでしまうパターン。

「保護を**追加する**変更」と「保護を**削除/縮小する**変更」はリスクの性質が非対称で、後者の方が見落とされやすい。本ドキュメントはその構造的な理由と、再発防止のための着眼点を一般化してまとめる。

排他ロックに限らず、null チェック・リトライ・タイムアウト・権限チェック・整合性ガード等、「**保護のために存在するコード**」全般に当てはまる。

## 症状

- 性能改善・コード簡素化のリファクタPRがマージされた直後〜数日後から、特定処理が**散発的に**失敗するようになる
- 単独実行・低負荷では再発しない。フルスイート並列実行・本番負荷で稀に発火
- エラーは「データが見つからない」「整合性違反」「期待値不一致」など、競合相手・タイミングによって症状がバラつく
- ログに、当該処理と同時刻に共有リソースを書き換えた別アクターの痕跡が残る

## 典型パターン（一般化）

ある処理 `T` が、3つの**異なる**共有リソース `resourceA` / `resourceB` / `resourceC` に対する保護を一括取得していた（同じリソースに Exclusive と Shared を同時取得しているわけではない点に注意）:

```
T:
  acquire Exclusive(resourceA)   // T が resourceA を書き換えるため
  acquire Shared(resourceB)      // T が resourceB を読むため、他者の書き換えから守る
  acquire Shared(resourceC)      // T が resourceC を読むため、他者の書き換えから守る
  ...
  perform T
  release all
```

注意: モード（Exclusive/Shared）は **その処理 T がそのリソースをどう扱うか** を表すだけで、リソース自体に「読み専用」「書き専用」という固定的な性質があるわけではない。同じ `resourceB` に対して、別の処理 `U` は `Exclusive(resourceB)` を取って書き換える、ということは普通にある。だからこそ T は `Shared(resourceB)` で自分の Read を守る必要がある。

うち `Exclusive(resourceA)` の保持時間が長い／取得が他処理を詰まらせる、という観測から、Phase 分割で保持時間を短縮するリファクタを実施:

```
T:
  // Phase 1: resourceA を書かない重い前処理
  acquire Shared(resourceC)       // ← Shared(resourceB) が抜け落ちた
  perform setup                    //   （Phase 1 内で resourceB を読む経路があるのに）
  release Phase 1 locks

  // Phase 2: 短い検証
  acquire Exclusive(resourceA)
  acquire Shared(resourceB)
  perform verification
  release all
```

PR 説明とコミットメッセージは「Phase 1 では `resourceA` を**書き込まない**から `Exclusive(resourceA)` は不要」という正しい論証だったが、**`Shared(resourceB)` が Phase 1 から消えたことの根拠は議論されなかった**。

実際には Phase 1 内で呼ばれる下位処理が `resourceB` を**読む**経路を持っており、他アクター（`U` など）が `Exclusive(resourceB)` を取って `resourceB` を書き換える瞬間とレースして「データが見つからない」系のエラーで失敗する。

## 根本原因：保護の非対称性

### 「足す変更」と「外す変更」のリスク非対称

| 操作 | 失敗時の症状 | 検出経路 |
|---|---|---|
| 不要な保護を足す | 性能劣化（観測可能） | 計測・プロファイル |
| 必要な保護を外す | 散発的不整合（観測困難） | 確率的失敗・運次第 |
| 不要な null チェック | デッドコード | 静的解析 |
| 必要な null チェックを外す | 本番 nil 参照 | 本番事故 |
| 不要なリトライを足す | 遅延 | レイテンシ |
| 必要なリトライを外す | 散発的失敗 | エラーレート |
| 不要な mutex を足す | スループット低下 | プロファイラ |
| 必要な mutex を外す | データ競合 | 散発的不整合 |
| 不要な権限チェックを絞る | 機能不全（即発覚） | E2E |
| 必要な権限チェックを緩める | セキュリティ穴（潜在化） | 監査・侵害後 |
| 不要なトランザクションを足す | デッドロック・遅延 | 性能監視 |
| 必要なトランザクションを外す | 部分更新による不整合 | データ監査 |

共通する構造:

- **追加の失敗は「正常系で目に見える」** → 動作確認で即落ちる
- **削除の失敗は「異常系・並行系で目に見えない」** → 確率的にしか発火せず、レビュー・CI を素通りする

「行を増やす変更」より「行を減らす変更」の方が、レビュアーの視野から影響範囲が見えにくい（diff 上は単に "1行減った" だけ）。

### Read 経路が見落とされる（無言の波及削除）

排他制御削除リファクタの典型的な誤論証パターン:

> 「Phase 1 は `resourceA` を書き込まないから `Exclusive(resourceA)` は不要」 → 正しい  
> （diff 上で同時に `Shared(resourceB)` も消えているが、根拠は記載されない）→ **無言の波及削除**

1つのロック削除に正当な根拠を与えると、その「同じ PR で行われた他のロック削除」もレビュアー・自動レビューの視野から漏れやすい。書かれた論証は `resourceA` のことしか述べていないのに、`resourceB` の削除がそれに紛れて通る。diff 上は両方とも単に「行が消えた」状態で、見た目から区別できない。

注: 同じリソースに対して `Exclusive` と `Shared` を同時に保持することは（reader-writer ロックの定義上）不可能なので、「同一リソースで Exclusive を外したついでに Shared も外す」という誤論証は構造的に発生しない。問題は常に**異なるリソース間**で起きる。

排他ロックの基本原則:

```
Exclusive を必要とする条件: 自分が書く
Shared を必要とする条件:    自分が読む（他者の書き換えから自分の Read を守る必要がある）
```

`Shared(resourceB)` は「他者が `Exclusive(resourceB)` を取って `resourceB` を書き換える瞬間から、自分の Read を守る」ためにある。よって `Shared(resourceB)` の要否は、**自分が `resourceB` を読むかどうか**で決まり、`resourceA` への書き込み有無とは独立。

Phase 内で呼ばれる下位処理（RPC、共通ヘルパ、共有ライブラリ等）が**間接的に対象リソースを Read する経路**は、実装を読み下さないと見えないことが多い。書き込みパスのみを根拠に削除すると Read 側の保護が抜け落ちる。

これは排他ロック以外でも同型:
- 「関数 X が例外を投げないから X 用の try/catch は不要」と説明する流れで、無関係な「関数 Y 用の try/catch」も diff 上で一緒に消えるが Y の議論は無い、というパターン
- 「このリトライは原因が解消したから不要」と説明する流れで、別の原因に対する別のリトライも巻き添えで消える、というパターン

論証で言及されたものと、diff で削除されたものが**集合として一致しているか**を毎回チェックする必要がある。

### 検出ガードレールが「たまたま」緑

このタイプのバグは特定条件下でしか発火しない:

- 競合相手が、削除側 Phase の処理実行中ピンポイントに被る必要がある
- スケジューリング次第で発火率が変わる
- PR の検証では発火せず、別のタイミングで踏む

「テストが緑」「動作確認が通った」は十分条件ではないのに、最終ガードレールとして機能してしまう。**観測されない＝存在しない、ではない**。

### 自動レビューの限界

自動レビュー（AI、Linter、静的解析）は構文・型・既知パターンの問題を検出できるが、**動的な並行実行レース・タイミング依存の正しさは原理的に保証できない**。

実例として、AI レビューが該当箇所を「指摘っぽい」コメントで触れることはあるが:

- 「💡（参考）」扱いで blocker にならない
- AI 自身も確信を持てず、「コメントを書け」程度の体裁で核心を突かない
- 結果として人間レビュアーが unresolved のまま APPROVE → マージ

静的解析が `no new issues` と返答しても、それは**そのツールの検出範囲内で問題が無い**だけで、**並行実行の正しさは保証していない**。

### 規約・ガイドラインが「追加方向」しか縛っていない

多くのプロジェクト規約は「保護を**追加する**前に対応コードを読め」と書いてあっても、「保護を**削除する**前に読め」とは書いていない。追加方向しか縛っていないと、削除方向は規約の穴になる。

レビューガイドラインも「新機能追加時のチェックリスト」は整備されていても、「既存保護の削除/縮小時のチェックリスト」は無いことが多い。

### Phase 分割は Phase 間で保護が切れる

保持時間を短縮するため Phase に分割する場合、**Phase 間の隙間で他アクターが介入できる**点を意識する必要がある:

```
phase1: acquire / work / release
        ← この瞬間、他者が Exclusive を取って書き換えうる
phase2: acquire (Shared 含む) / verify / release
```

「Phase 1 で Shared を取った／Phase 1 を抜けたら Phase 2 でも保護が続く」と直感的に勘違いしやすい。実際には**各 Phase は独立して必要な保護をすべて取り直す**必要がある。

これは排他ロック以外でも同型: トランザクションを 2 つに分割すると、間にコミット境界ができて他者が割り込めるのと同じ構造。

### 生成された PR 説明が再検証インセンティブを下げる

PR 説明が綺麗な markdown で AI 生成・テンプレ生成されていると、レビュアーは「ちゃんと分析してある」という印象を受け、論証の前提（保護が何を保護しているか）を再検証するモチベーションが落ちる。**生成された綺麗さと正しさは独立**。

「Before/After 図がある」「数値計測が載っている」「論証が箇条書きで整理されている」は、**論証の前提が正しいことを保証しない**。前提部分（「`resourceA` を書き込まない」など）が事実かどうかは、実装を読まないと検証できない。

## 検出と予防

### 保護削除PR用チェックリスト

排他ロック・null チェック・リトライ・トランザクション・権限チェック等の**保護**を削除・縮小するPRでは、最低以下を resolve してからマージする:

1. **削除した各保護について、削除前にカバーしていた経路を列挙したか**
   - 書き込み経路だけでなく読み込み経路も全部
   - 内部で呼ぶ下位関数・RPC・共通ヘルパまで掘り下げる
2. **その全経路について、削除しても安全な根拠を明示したか**
   - 「キャッシュ経由で固定」「呼ばれない経路」「セッション内で immutable」など
3. **競合相手（同じリソースに書き込みうる別アクター）を網羅し、衝突しないことを確認したか**
   - grep で対象リソースに書き込むコードを全件洗い出す
4. **削除した各保護について、PR 説明・コミットメッセージに個別の根拠を書いたか**
   - 「2つの保護は説明したから3つ目もたぶん大丈夫」というレビュアーの認知バイアスを断つ
   - 「無言で消える保護」は将来の git log 調査でも理由が辿れず、後続開発者が「もともと不要だったのだろう」と誤解する
   - **削除した保護の数 == 個別根拠の数** が成立しているか確認する。不足している項目こそレビューの本丸
   - 「Phase X から `Lock(Y)` を外した。理由は Z」という形式で 1 削除 = 1 段落を徹底する
5. **削除と性能改善の効果が分離可能なら、削除は別PRに切り出したか**
   - 性能改善（Phase 分割など）と保護削除（Shared 撤去）は別の意図。混ぜると影響範囲が混在してレビューしにくい

### 規約の双方向化

プロジェクト規約に以下のような項目を追加:

> **MUST**: 排他制御・null チェック・リトライ・トランザクション境界・権限チェック等の保護を**削除/縮小**する場合は、対応する実装を読み、保護がカバーしていた経路を全て洗い出してから外す。書き込み経路だけを根拠に Read 側保護を削除しない。

### レビュー時の視野補正

- diff 行数が少ない・削除中心のPRこそ、影響範囲が見えにくいことを意識する
- 「保護を外す」差分はレビューで色付け・要注意ラベル等で視覚的に持ち上げる
- 自動レビューの `💡` 指摘でも「保護を外している部分への指摘」は ⚠️ 以上に格上げするヒューリスティクス
- PR 説明が AI 生成・テンプレ生成だった場合、論証の**前提**部分を独立に再検証する

### 検証手段の補強

- 並列ストレステスト（同じ処理を多重実行で繰り返し）を週次・夜次で走らせ、低発火率の Flaky を炙り出す
- 共有リソースへのアクセス・保護取得/解放の時系列ログを構造化して残し、Flaky 発生時に「同時に書き換えていたアクター」を逆引きできるようにする
- 本番環境では、競合発生時の指標（失敗率・retry 回数・待ち時間）を監視し、削除リファクタ後にズレていないか観察する

### 「保護を外す」と「性能改善」を分離する

ロック保持時間短縮のリファクタは、しばしば次の 2 つを同時に行う:

1. **保護の構造を変える**（Phase 分割、ロック粒度変更、保護対象の見直し）
2. **保護を一部削除する**（不要と判断したものを外す）

両者が混在した PR は、性能改善効果に注意が引かれて削除部分のレビューが甘くなる。可能なら:

- まず構造変更だけ行い、削除はせずに動作と性能を確認
- その後、各削除を独立した小さい PR として、根拠を明示してマージ

## 教訓

1. **保護を外す変更は、機能を足す変更と同等以上のレビュー深度が必要**。直感（小さい diff = 安全、行数削減 = 良い変更）と逆向き
2. **「書かないから不要」は排他ロックの場合 Exclusive にしか適用できない**。Shared は Read 保護のためにあるので、Read 経路を独立に検討する
3. **検証ガードレール（テスト・CI・動作確認）が緑であることは、保護削除の正しさの証明にはならない**。観測されないバグはタイミング次第で潜伏する
4. **Phase 分割・トランザクション分割は、Phase 間で保護が一旦切れることを意味する**。各 Phase は独立に必要な保護を取り直す
5. **自動レビューが弱い指摘しか出さなくても、保護削除に関する箇所は人間が掘る価値がある**。AI が確信を持てない領域こそレビューの本丸
6. **規約は「追加」だけでなく「削除/縮小」方向にも MUST を書く**。片方だけだと反対側が穴になる
7. **生成された綺麗な PR 説明と、論証の正しさは独立**。前提部分は実装を読んで再検証する
8. **「保護の構造変更」と「保護の削除」は別 PR に分ける**。混ぜると削除部分のレビューが甘くなる
9. **削除した保護の数 == 個別根拠の数 を成立させる**。一部の削除だけ説明され、他は無言で消えているなら、その「無言の削除」こそレビューで掘るべき箇所。git log にも理由が残らないため、将来の開発者が「もともと不要だったのだろう」と誤解する原因になる
