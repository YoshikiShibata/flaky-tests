# テストデータの有効期限が次の暦境界とちょうど一致するパターン

## パターン名

**StartTime が暦境界にアンカーされ、duration が次の境界に着地するテストデータ**

## 問題の本質

テストデータが次の 2 条件を同時に満たすと、深夜帯の実行で `ExpireTime` をまたいでテストが失敗する。

1. データの `StartTime` が**暦境界**（`00:00 JST` など）に固定される
2. データの寿命（`duration`）が暦境界の整数倍時間（24h、48h など）で、`ExpireTime` がちょうど**次の同じ境界**に着地する

テストの実行に数分以上かかる場合、`ExpireTime` 直前に開始したテストは実行中に境界をまたぎ、その時点でデータが API レスポンスから除外され、後段の検証が失敗する。

## 発生条件

以下の 3 条件がすべて揃った場合に発生する:

1. **データの `StartTime` が暦境界に固定**: ドメインのスケジューラ等が「曜日 + 時刻 0:00」のような形でアンカーする実装
2. **データの `duration` が境界の整数倍**: 24h、48h、72h など、暦境界に着地する値
3. **テスト実行時間 > 残り境界時間**: 例えば 23:51 JST にテスト開始、テスト実行 9 分超 → 00:00 JST を跨ぐ

加えて、対象 API が「期限切れ × 未達成 = 非表示」のような可視性フィルタを持っていると、データが消失する形で失敗する。

## 実例（抽象化）

時限性リソース（クーポン、キャンペーン、タスク、抽選券など）を生成するヘルパーが、内部で StartTime を「本日の暦境界」に固定し、`duration` パラメータで寿命を設定するパターンを想定する。

```go
// 時限リソースを生成
helper.createResource(t, resourceConfig{
    durationHours: 24, // ← 暦境界の整数倍
    // ...
})
// 内部仕様: StartTime = 本日 00:00 JST, ExpireTime = StartTime + durationHours
//          → ExpireTime = 翌日 00:00 JST

// ... 検索・状態遷移などの操作が続く ...

// 1 回目のリソース取得（テスト開始 5 分後、まだ昨日 23:55 JST）
res := client.ListResources(ctx) // リソース見える ✅

// 2 回目のリソース取得（テスト開始 10 分後、00:00 JST を跨いだ）
res = client.ListResources(ctx) // リソース消える ❌
// failure: Resource with ID "..." not found in response
```

**失敗ログの典型形:**

```
YYYY/MM/DD 00:00:0X ...
<test_file>:<line>: <Resource> with ID "..." not found in <ListResponse>
--- FAIL: <TestName> (XXX.XXs)
```

特徴:
- テスト所要時間が数百秒
- 失敗時刻が `00:00` JST 付近
- 失敗内容が「データが見つからない」「リストが空」

## 対策パターン

### 対策 1: duration を境界からずらす（推奨）

最もシンプル。duration を「暦境界の整数倍」から外す。

```go
// Before（脆弱）
durationHours: 24, // ExpireTime = 翌日 00:00 JST

// After（堅牢）
// StartTime は本日 00:00 JST に固定されるため、durationHours=24 だと
// 深夜実行時に ExpireTime（翌日 00:00 JST）を跨ぐ flakiness がある。
// 48h にすることで深夜実行でも 24 時間以上の余裕を確保する。
durationHours: 48, // ExpireTime = 翌々日 00:00 JST（24h+ の余裕）
```

複数 duration を sort 順序検証で使う場合は、相対順序を保ったまま全体シフト:

```go
// Before: 24 / 48 / 72
durationHours: 24, // 最短値が脆弱
durationHours: 48,
durationHours: 72,

// After: 48 / 72 / 96
durationHours: 48, // 全値が安全
durationHours: 72,
durationHours: 96,
```

### 対策 2: duration を奇数時間に（境界に着地させない）

例えば `durationHours: 36` でも、`ExpireTime = 翌日 12:00 JST` となり昼間に着地する。深夜帯と離れていれば flakiness が起きにくい。

### 対策 3: StartTime を境界からオフセット（要ドメイン実装変更）

StartTime の決定ロジックを「本日 00:00」ではなく「`time.Now()` 基準」に変更する。ただしこれはテストヘルパーやドメイン実装の改修が必要なため、コストが高い。

### 対策の優先順位

| 優先度 | 対策 | コスト |
|---|---|---|
| 1 | duration を境界からずらす（24→48 等） | 数値 1 つの変更 |
| 2 | duration を奇数時間にする | 数値変更 + 期待値再計算 |
| 3 | StartTime のアンカリングを変える | ドメイン実装/ヘルパー改修 |

## 一般化した教訓

1. **暦境界 + 暦境界の整数倍 duration の組み合わせは避ける**: テスト実行時間を超える余裕を必ず確保する
2. **「ちょうど」な値を使わない**: `durationHours: 24` のような「日数」を表現する整数は境界に着地しやすい。実装上の最小限を守りつつ、テストでは安全側に倍以上の値を使う
3. **CI 実行時刻はランダム**: 深夜帯にテストが実行される可能性を常に考慮する
4. **テスト所要時間より境界までの残り時間が短い帯がある**: 例えばテスト所要時間 10 分なら、`境界 - 10 分以降〜境界` の時間帯はリスク帯
5. **データ寿命が暦単位（24h、月、年）の duration を持つテストは特に要注意**: 暦境界と必ず同期する

## 検出方法

「ある時間帯（特に 23:00 JST 以降）にだけ失敗する」テストはこのパターンを疑う:

- 失敗時刻が `00:00 JST` 付近
- 失敗内容が「データが見つからない」「リストが空」など
- データの寿命が `24h`、`48h` などの暦境界整数倍

duration 設定値で grep を掛けると、類似の設定を持つテストを一括で洗い出せる。
