# テスト用DBエミュレータと本番DBエンジンの挙動差で表面化する Flaky Test

## はじめに

ローカルや CI で使うテスト用の DB エミュレータは、本番のマネージド分散 DB を
完全に再現しているわけではありません。とくに **行順序の保証** と
**トランザクション分離レベル（同時実行制御）** は、エミュレータの方が緩い
（または挙動が異なる）ことがあります。

その結果、「エミュレータでは安定して通るのに、本番相当の DB エンジンを使う環境でだけ
非決定的に落ちる」フレーキーテストが生まれます。エミュレータの緩い挙動が
**潜在的な非決定性を覆い隠していた**ために、コードもテストもその非決定性に
気付かず依存してしまうのが本質です。

本記事では、このカテゴリの実例を2つ（順序保証・分離レベル）紹介し、
一般的な対策をまとめます。

## 問題の本質

テストが暗黙に「ある DB 実装の都合の良い挙動」に依存していると、
別の DB 実装（本番エンジン）に載せ替えた瞬間に前提が崩れる。

```
エミュレータ:  緩い挙動  → 偶然テストが通る（非決定性が顕在化しない）
本番DBエンジン: 厳密な挙動 → 非決定性が顕在化し、実行ごとに結果が揺れる
```

依存しがちな「都合の良い挙動」の代表が次の2つ。

1. **行順序**：`ORDER BY` の無いクエリの返却順がたまたま安定している
2. **同時実行**：並行トランザクションの可視性・直列化のタイミングがたまたま都合よく揃う

---

## 事例1：ORDER BY の無いクエリと固定順アサーション

### 背景

あるレスポンスに含まれる「ID のリスト（繰り返しフィールド）」を、
テストが **固定順のスライス**で期待値比較していた。

```go
want := &Response{
    IDs: []string{"AAA", "BBB"}, // この順序を期待
}
// ...
if diff := cmp.Diff(want, got); diff != "" {
    t.Errorf("mismatch:\n%s", diff)
}
```

このリストの生成元クエリには **`ORDER BY` が無かった**。

```sql
SELECT id FROM items
WHERE category = @category AND platform = @platform
-- ORDER BY が無い → 返却順は保証されない
```

### 失敗メカニズム

- エミュレータでは返却順がたまたま安定していて、テストは常に PASS
- 本番 DB エンジンでは行順序が保証されないため、実行ごとに順序が入れ替わる

```
get:  ["BBB", "AAA"]   ← 順序が反転
want: ["AAA", "BBB"]
→ 差分が出て FAIL
```

複数要素を返す**他のケースも同じ構成なら潜在的に flaky**である点に注意
（今回偶然落ちたケースだけの問題ではない）。

### 対策

- **テスト**：繰り返しフィールドを**順序非依存**で比較する。
  多くのアサーションライブラリに「繰り返し要素をソートしてから比較する」
  オプションがある（例：protobuf 比較の repeated フィールドソート、
  スライスソート比較）。順序に意味が無いフィールドにはこれを使う。
- **本番コード**：順序に意味があるなら、クエリに決定的な `ORDER BY` を付ける
  （タイブレークまで一意になるよう、必要なら複合キーで）。

順序に意味が無い（集合として扱う）なら**テスト側を順序非依存にする**のが筋。
順序に意味があるなら**本番クエリを決定化**する。どちらかであって、放置は不可。

## 事例2：並行入金に依存した「残高」がタイミングで枯渇する

### 背景

各エンティティが「消費可能な残高（リソース）」を持ち、
ある操作はその残高を**消費（デビット）**する。残高が不足すると
前提条件エラーで操作が失敗する。

テストのセットアップが、多数の操作を**並列**で実行していた。
あるエンティティは、

- 初期残高が**わずか数回分**しかない
- にもかかわらず、そのエンティティが**多数回（初期残高をはるかに超える回数）消費**する

という構成になっていた。不足分は、**別の並行操作がそのエンティティへ
残高を入金（クレジット）する**ことで埋められる前提だった。

```
あるエンティティの収支:
  初期残高:        2回分
  並行クレジット:  上限つきの報酬 + 他操作からの副次的な入金（タイミング不定）
  必要なデビット:  20回分
→ クレジットがデビットに「間に合えば」成立する、というタイミング依存
```

### 失敗メカニズム

- **エミュレータ**：分離が緩く、並行トランザクションが互いの未コミット状態を
  跨いで進む／上限チェックが甘くなる等で、入金が過剰気味・早めに反映され、
  残高が枯渇せず PASS
- **本番 DB エンジン**：直列化が厳密で、入金トランザクションが正しく順序化される。
  その結果、あるデビット時点でまだ必要なクレジットがコミットされておらず、
  残高ゼロ → 前提条件エラーで FAIL

エラーは「残高不足」だが、同じエラーコード／メッセージが
**別の前提条件（回数上限など）と共用**されていると、原因の切り分けが難しくなる
（メッセージだけでは真因が判別できない）。

### 対策

- **テストのリソースをタイミングから切り離す**：相互入金の到着順に依存させず、
  セットアップ段階で各エンティティに**十分な初期残高を直接付与**しておく。
  「正規 API では作りにくい初期状態」を作るための DB ヘルパー直接書き込みは、
  **理由コメント付き**で使うのがよい。
- 付与量は、上限（容量）を超えない範囲で、消費回数に十分な余裕を持たせる
  （上限に張り付くと、別の「満タン時スキップ」分岐に入って挙動が変わることがある）。
- **インタラクション回数（=検証対象の集計）が変わらない**ことを確認する。
  残高を底上げしても操作の発生回数は変わらないので、件数アサーションは不変のはず。
- リソース消費が発生しないケースには付与をスキップするガードを入れて無駄を省く。

### 補足：これはテスト設計の脆さでもある

「初期残高が足りないが、並行入金が間に合うはず」という設計は、
DB の分離レベルというインフラ実装に暗黙依存している。
テストの安定化（残高の事前付与）が現実的な落としどころだが、
本質的には「相互作用のタイミングに結果が依存する」セットアップ自体が脆い、
という認識を持っておく。

---

## 一般的な対策（まとめ）

1. **順序保証に依存しない／決定化する**
   - 集合的フィールドは順序非依存比較。順序に意味があれば一意な `ORDER BY`。
2. **同時実行のタイミングに依存しない**
   - テストの前提リソースは、並行入金の到着順に賭けず事前に確定させる。
3. **エミュレータと本番の差分を疑う**
   - 「ローカル／エミュレータでは再現しないが CI（本番相当 DB）で落ちる」flaky は、
     まず**行順序保証**と**分離レベル**の差を疑う。
4. **共用エラーに注意**
   - 複数の前提条件が同一エラーコード／メッセージを共有していると真因がぼやける。
     切り分けには発生条件（定数・上限値）を実際に当てはめて潰す。

## チェックリスト

- [ ] レスポンスやリストを**固定順**でアサートしているが、生成クエリに `ORDER BY` が無い
- [ ] 繰り返し（配列）フィールドの**順序に意味が無い**のに順序まで比較している
- [ ] テストのセットアップで、エンティティの**残高／カウンタ**を並行操作で増減させている
- [ ] あるエンティティの**初期値が、必要な消費回数に対して不足**している
- [ ] 不足分が「他の並行操作からの入金が間に合うこと」に暗黙依存している
- [ ] 「ローカル／エミュレータでは通るが CI（本番相当 DB）でのみ落ちる」flaky がある
- [ ] 失敗時のエラーコード／メッセージが**複数の前提条件で共用**されている

## 教訓

1. **エミュレータは本番 DB の近似でしかない** — 行順序保証と分離レベルの差は、
   緩い側で潜在バグを覆い隠す。
2. **非決定性は「テストが依存していい挙動か」で線引きする** — 順序や並行入金の
   タイミングは、明示的に決定化しない限り依存してはいけない。
3. **テストの前提は「賭け」にしない** — 必要な初期状態は、相互作用のタイミングに
   頼らず確定的に用意する。
4. **本番でのみ落ちる flaky は DB 実装挙動を最初に疑う** — 再現環境が無くても、
   順序保証・分離レベルの観点から机上で原因を特定できることが多い。
