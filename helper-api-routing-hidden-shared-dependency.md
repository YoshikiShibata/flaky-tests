# テストヘルパーの「直接DB → API/usecase経由」化が生む隠れた共有依存とロック抜けフレーク

## 一言サマリ

テストヘルパーの内部実装を「素の DB アクセス」から「本物の RPC / usecase 経由」に変えると、
そのヘルパーは usecase が読む**共有リソース（マスタデータ等）への暗黙の依存**を新たに獲得する。
呼び出し側はこの依存に気づかず以前と同じ前提（ロックなし等）で呼び続けるため、
**並列実行下で共有リソースを書き換える別テストと競合し、確率的フレーク**になる。

「忠実度を上げる良いリファクタ」が、同時に「結合度と暗黙の前提条件を増やす副作用」を持つ、という構造。

---

## 症状 / シグネチャ

- 低頻度で落ちる（再実行で復活する）テスト。
- エラーが **`... is not found` 系を内部要因で返す Internal/500**。
  例: `failed to findXxxSetting(): xxx setting is not found` のような「あるはずのマスタが無い」系。
- 失敗するのは「書き換える側」ではなく、**ただ読んでいるだけのテスト**。
- 「テストもインフラも古いのに、最近になって落ち始めた」——直近に**ヘルパー / 読み取り経路のリファクタ**が入っている。
- 失敗箇所のスタックや行が、一見その共有リソースと無関係に見える（`createXxx` `findXxx` 等の名前のヘルパー内部で起きる）。

---

## 根本メカニズム（一般化）

フレークが成立するには **2つの前提**が同時に必要。

1. **共有リソースに「一瞬消える / 不整合になる窓」がある**
   - 例: グローバル共有のマスタ行を、あるテストの setup/cleanup が「全削除 → 再投入」する。
   - その間、行が存在しない瞬間が生まれる（多くは排他ロック下だが、**ロックは取得者同士しか排他しない**）。

2. **その共有リソースを、ロック/同期なしで読む経路がある**
   - ここに「ヘルパーの API 経由化」が効く。**以前は共有リソースに触れなかったヘルパー**が、
     usecase 経由化によって「共有リソースを読む」コードパスを内部に獲得する。
   - 呼び出し側は内部実装の変化を知らないので、同期（ロック取得）を追加しない。

→ ①の窓の最中に②の読みが走ると、不整合（行欠落）を観測して失敗する。

### なぜ②が「ヘルパーの API 経由化」で生まれるか

| | Before（直接DB） | After（API/usecase経由） |
|---|---|---|
| 読む対象 | 目的の1テーブル/1行だけ | usecase が辿る**全依存**（マスタ・他テーブル・整合性チェック・派生計算） |
| 暗黙の前提 | ほぼ無し | 「依存リソースが整合した状態で存在する」 |
| 呼び出し側の同期義務 | 無し | **本来は共有リソースのロックが必要**（が、見えないので忘れられる） |

ヘルパー名（`createXxx`, `findXxxById` 等）からは「どの共有リソースを読むか」が分からないため、
**内部実装を1箇所変えるだけで、そのヘルパーを呼ぶ全テストの前提が一斉に変わる**のが怖い点。

---

## なぜ見つけにくいか

- **コンパイルも単体実行も通る**。タイミングが揃ったときだけ落ちる確率的フレーク。
- リファクタ PR の CI が一度通っても、後日**別の並列負荷**で初めて顕在化する（潜伏する）。
- 依存が呼び出し側のコードから見えない（ヘルパーの内部に隠れる）。
- 失敗の確率は「共有リソースを書き換える並列テストの数 × 並列度」に比例して上がるため、
  関係ない別テストの追加が引き金に見えることもある。

---

## 検出方法（横断チェックの手順）

「共有リソースを起点にした逆引き」が効く。

1. **共有リソースを読む usecase / RPC を列挙する**
   - 例: 該当マスタを読む fetcher（`XxxFetcher.Get` 等）の**呼び出し元メソッド**を grep。
   - private ヘルパー経由の場合は**コールグラフを1段たどる**（呼び出し元の RPC まで特定）。

2. **その RPC を叩く e2e ヘルパー / クライアント関数を特定する**
   - クライアントを直接呼ぶ箇所だけでなく、
     **名前からは読み取りと分からないヘルパー経由の間接呼び出し**（例: ある操作系ヘルパーが内部で読み取り API を呼ぶ）も洗う。← ここが漏れやすい

3. **各 Test 関数で、その読み取り呼び出しを「カバーするスコープ」で同期（ロック）を取得しているか判定**
   - クラス(a): ロックを**全く取らない**テスト。
   - クラス(b): ロックを**サブテスト内で取得しているが、読み取り呼び出しがそのスコープの外**にある。← 静的に見落としやすい

4. **誤検知を usecase コードで潰す**（重要）
   - 「一覧系 RPC だから読むだろう」と仮定しない。**実際にその RPC が共有リソースを読むか**をコードで確認。
   - 条件付き読み取り（空リストならループに入らず読まない / feature flag / 実行モードで skip）は実害なしのこともある。

> 並列 Explore エージェント等でファンアウトする場合も、**最後に usecase コードで読み取り有無を必ず裏取り**する。
> エージェントは「一覧系=読む」と過剰報告しがち。

---

## 修正パターン

- **第一選択: 読み取り側に同期（共有ロックの取得）を追加**して、書き換え窓と排他する。
  - 読むだけなら共有(Shared)ロック、書き換えるなら排他(Exclusive)ロック。
  - クラス(b) は「読み取り呼び出しをロック済みスコープの中に入れる」（サブテストで包む等）。
- **代替/補強: 共有リソースの書き換えを非破壊にする**
  - cleanup を「全削除 → 再投入」ではなく **upsert（行を消さない）** にすれば、そもそも欠落窓が消える。
  - ただし共有インフラの変更は影響範囲が広いので慎重に。
- **プロダクション側は基本いじらない**
  - 「依存リソースが無い＝データ不整合」で Internal を返すのは、異常を握りつぶさず明示的にエラーにする方針であれば正しい挙動。
    フレークの原因は**テストの同期不足**であって本番バグではない、を切り分ける。

## 予防 / チェックリスト

ヘルパーを「直接DB → API/usecase経由」にリファクタするときは:

- [ ] その usecase が読む**共有リソース（マスタ等）を洗い出した**か？
- [ ] それらの共有リソースに対応する**ロック/同期を、ヘルパーの呼び出し側へ伝播**したか？
      （リファクタ PR の差分に「読み取り経路の変更」と「同期の付与」を**セットで**入れる）
- [ ] 共有リソースを読む RPC / ヘルパーの **doc コメントに「必要なロック」を明記**したか？
      （呼び出し側が前提を見落とさないように）
- [ ] そのヘルパーを呼ぶ**既存テストを横断チェック**したか？（内部実装変更は全呼び出し側の前提を変える）
- [ ] 共有リソースの setup/cleanup が**破壊的（削除窓を作る）**でないか確認したか？
