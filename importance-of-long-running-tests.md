# 長時間ランニングテストの重要性

## テストの限界

テストは「バグの存在」を証明できるが、「バグの不在」は証明できない。

- 100回PASSしても、101回目に失敗する可能性がある
- 全てのテストが成功しても、未発見のバグが存在しないことの証明にはならない

## なぜ長時間ランニングテストが重要か

### 1. 低確率の問題を発見できる

| 失敗確率 | 発見に必要な実行回数（期待値） |
|---------|------------------------------|
| 10%     | 約10回                        |
| 1%      | 約100回                       |
| 0.1%    | 約1000回                      |

実例: 14回目で初めて失敗したテスト → 発生確率は約7%程度

### 2. 様々なタイミングをカバーできる

並列テストの実行順序やタイミングは毎回異なる：
- テストAとテストBの開始タイミングの差
- データベースへのアクセス順序
- クリーンアップ処理のタイミング

多くの実行を重ねることで、より多くのタイミングパターンを試すことができる。

### 3. 環境の変化を捉えられる

時間帯や状況によって異なる条件：
- CI環境の負荷状況
- リソース競合の程度
- ネットワーク遅延の変動

長時間実行することで、これらの変動要因を網羅できる。

### 4. 修正の検証にも有効

修正後も繰り返し実行することで：
- 修正が効果的かを確認できる（ただし証明ではない）
- 新たな問題を引き起こしていないかを確認できる

### 5. テスト分割 (sharding) で隠れる cross-test 干渉を発見できる

CI では実行時間短縮のため、test suite を複数 shard に分割して並列実行することが多い。各 shard が独立した DB / 環境で動作する構成では、**shard 跨ぎのテスト間干渉は CI では原理的に検出できない**：

- shard A の test X が DB に「グローバル可視データ」(全 user に見える record / 全 product に効く設定 等) を作成する
- shard B の test Y が「自分以外は同種データを作らない」前提で assertion
- CI: X と Y は別 shard で別 DB → 干渉せず両 PASS
- ローカル全テスト一括実行: X と Y が同一 DB を共有 → Y が干渉を受けて FAIL

**重要**: この種の問題は、既存セクション 1〜4 の **「同じテストを繰り返し実行する」アプローチでは解決しない**。CI 上で shard 分割実行を 100 回、1000 回繰り返しても、shard isolation がある限り **shard 跨ぎの干渉は原理的に検出不可能**。「1 回でも良いから全テストを 1 つの DB で一括実行する」ほうが検出力が高い、というのが既存セクションとは異なる本セクションの主張。

#### 典型的な失敗パターン

| パターン | 例 |
|---|---|
| グローバル broadcast データの作成 | 「全ユーザ対象」フラグ / wildcard target を持つレコード |
| 共有設定の上書き | feature flag、 system parameter、ranking master データ |
| seed データの追加 / 削除 | 各テスト独自の category / tag を全 user に見える形で追加 |
| 時刻ベースの side effect | 「今日アクティブな」を変更するレコードを past/future に作る |

#### Production 動作との類似度

- **production**: 単一 DB × 多数の concurrent user → ローカル全テスト一括実行と類似
- **CI shard**: 単一 DB × 一部テスト (分割実行) → production と非類似

ローカル全テスト一括実行のほうが production の負荷パターンに近く、本番で発生し得る race condition / 干渉を再現しやすい。

#### 顕在化の遅延リスク

新しいテストが「グローバル可視データを作る」ものとして追加された瞬間、**理論的には**既存テストが flaky 化条件を満たす。しかし:

- CI shard 分離で見えない → merge 時点で気づかれない
- ローカル全テスト実行で初めて顕在化 → **テスト追加から数日〜数週間後に発覚**
- 原因の特定が困難 (どの追加が引き金か特定するのに git log 漁りが必要)

#### 検出のために

- **shard を増やすと検出力が下がる**: CI 並列度を上げるほど、shard 内テスト数が減り、cross-test 干渉の発見機会が消える
- **定期的なローカル全テスト実行が補完手段**: 開発機の遊休時間に「全テスト 1 回」を回すだけでも検出可能
- **テスト追加時のセルフチェック**: 「このテストはグローバルに見えるデータを作るか?」を確認し、作る場合はコメントで明示する

## PRでのテスト実行だけでは不十分

一般的な開発フローでは、PRを作成するとCIでテストが1回だけ実行される。しかし、これだけではFlaky testの発見には不十分である。

| 失敗確率 | 1回の実行で発見できる確率 |
|---------|-------------------------|
| 10%     | 10%                     |
| 1%      | 1%                      |
| 0.1%    | 0.1%                    |

**問題点**:
- 1回のテスト実行でPASSしても、Flaky testが潜んでいる可能性は高い
- マージ後に他の開発者のPRで突然失敗し始め、原因特定が困難になる
- 「自分のPRでは成功していたのに」という状況が発生する

**PRでのテスト実行の位置づけ**:
- 明らかなバグ（100%失敗するテスト）の検出には有効
- Flaky test の検出には不十分
- PRのテストがPASSしたからといって、問題がないとは言えない

したがって、PRでのテスト実行とは別に、長時間ランニングテストを実施する必要がある。

## 実践的なアプローチ

### 開発マシンでの長時間実行（推奨）

開発者が開発マシンを使っていない時間（夜間や休憩中など）に繰り返しテストを実行する方法が最もコスト効率が良い。

```bash
# 例: 100回繰り返し実行（夜間に実行）
for i in {1..100}; do
  echo "Run $i"
  go test -tags=e2e ./... || echo "Failed at run $i"
done
```

**メリット**:
- GitHub Actions などの CI サービスの実行時間コストがかからない
- 開発マシンの遊休リソースを有効活用できる
- 好きなだけ繰り返し実行できる（コストを気にせず）
- ローカル環境での問題も発見できる

**実行タイミングの例**:
- 夜寝ている間
- 昼休み中
- ミーティング中
- 週末

### CIでの継続的な実行

CI での実行はコストがかかるため、開発マシンでの実行を補完する位置づけとする。

- 定期的（例: 毎日1回）に全テストを実行するジョブを設定
- 失敗時に通知を送信
- 失敗ログを保存して後から分析できるようにする

**注意**: GitHub Actions などの従量課金制 CI サービスでは、長時間ランニングテストを頻繁に実行するとコストが膨らむため注意が必要。

### 失敗パターンの記録

- いつ失敗したか
- 何回目で失敗したか
- エラーメッセージの内容
- 同時に実行されていた他のテスト

## 結論

長時間ランニングテストは「問題がない」ことを証明することはできないが、以下の点で価値がある：

1. **発見**: 低確率のFlaky testを発見できる可能性が高まる
2. **信頼性向上**: より多くの条件でテストすることで、信頼性が向上する
3. **早期発見**: 本番環境で問題が発生する前に発見できる

「証明できない」からといって「無意味」ではない。確率的なアプローチとして、長時間ランニングテストは重要な品質保証手段である。
