# keyset ページネーションの tie-breaker 欠如による行の取りこぼし

## はじめに

「間欠的に失敗する E2E テスト」は、真っ先に *テスト側のフレーク* を疑いがちです。
再実行して通れば「環境要因」で片付けてしまうこともあります。

しかし、間欠失敗が **サービス側（プロダクションコード）の実装バグ** の唯一のシグナル
であることがあります。本記事はその一例で、**keyset（カーソル）ページネーションが
ソートキーの同値（tie）を考慮していなかった** ために、条件次第で行を静かに取りこぼす
という、本番でも起き得るバグでした。テストはフレークだったのではなく、
並行性に敏感な実バグを正しく検出していたのです。

## 問題の本質

keyset ページネーションは、前ページ末尾の値を cursor にして次ページを引きます。
ソートキーが **一意でない** 列（例: `CreateTime`）のときに、cursor を
**strict な不等号（`<`）だけ**で構成すると、**同値の行がページ境界をまたいだ瞬間に
永久に脱落**します。

```
対象行（CreateTime DESC 順、1ページ = 3件とする）:
  行1  CreateTime=T3
  行2  CreateTime=T2
  行3  CreateTime=T1   ← ページ1の末尾（cursor = T1）
  ------------------- ページ境界 -------------------
  行4  CreateTime=T1   ← 行3と同値！
  行5  CreateTime=T0

ページ1: WHERE ... ORDER BY CreateTime DESC LIMIT 3
         → [行1, 行2, 行3]、cursor = T1
ページ2: WHERE CreateTime < T1 ...        ← strict `<`
         → [行5]                            ← 行4(=T1) が脱落！
```

行4 はページ1（LIMIT に収まらず）にもページ2（`< T1` で除外）にも現れず、
**どのページを走査しても返ってこない** 状態になります。

同じことは重複（同じ行が複数ページに現れる）としても起こり得ます。
本質は **「ソート順が同値行に対して非決定的で、かつ cursor がその同値を跨げない」**
という点にあります。

## 実例

### 背景

あるエンティティの一覧を返す API（1ページ 10 件）の keyset ページネーションが、
`CreateTime` だけを cursor にしていました。テーブルは所有者・ステータス・作成時刻で
引く構成です。

```sql
-- 実際に使われていた（tie-breaker 無し）クエリ
SELECT ...
FROM Records@{FORCE_INDEX=RecordsByOwnerIDStatusCreateTime}
WHERE OwnerID = @ownerID AND Status = @status
  AND CreateTime < @createTime          -- strict `<`、tie-breaker 無し
ORDER BY CreateTime DESC                  -- 第2ソートキー無し
LIMIT @limit
```

usecase 側は「前ページ末尾の `CreateTime`」を次 cursor にしていました。

### 症状

あるセットアップ処理が、この一覧 API を **全ページ走査** して特定のレコードを探し、
見つけたレコードに対して後続処理を行っていました。レコードは確実に作成済みなのに、
全ページを走査してもそのレコードが返ってこず、`not found` になって処理が失敗する、
という形で表面化しました。

### 根本原因

- `CreateTime` はアプリが `time.Now()`（言語ランタイムの壁時計）で設定していた。
  この壁時計の **分解能は実行環境によっては非常に粗く**、連続した `time.Now()` が
  **同一の値を返す**（`time.Now() == time.Now()` が true になる）。実測した環境では、
  連続呼び出しの約 26%、タイトなバーストでは約 95% が同一ナノ秒値だった。
  そのため同一所有者に対して短時間に複数レコードを作ると `CreateTime` が
  **同一ナノ秒値で衝突**する。
- **DB（Spanner）は丸めていない**。Omni / エミュレータ双方で TIMESTAMP 列は
  ナノ秒まで保存されることを実測で確認した（「DB がマイクロ秒に丸めるから衝突する」の
  ではなく、**アプリが渡す値がそもそも粗い時計に由来して同値**であり、DB はそれを
  忠実に保存している）。
- インデックス `(OwnerID, Status, CreateTime DESC)` にも一意列（主キー）が含まれず、
  同値行の順序は非決定的。
- その結果、同一 `CreateTime` のレコードがページ境界（10 件目 / 11 件目）をまたぐと、
  `CreateTime < cursor` の strict 比較で境界外の行が脱落し、一覧 API がその
  レコードを一切返さなくなる。

### なぜ「間欠的」だったか

脱落が起きるには **(1) 同一所有者のレコードが複数ページ分あり、(2) そのうち2件以上が
同一 `CreateTime` を持ち、(3) その同値がちょうどページ境界をまたぐ** という3条件が
同時に成立する必要があります。`CreateTime` の衝突は「粗い壁時計が同一値を返す」という
タイミング依存なので、並行負荷が高いときほど起きやすく、実行のたびに結果が揺れました。

つまり **テストのフレークではなく、並行タイミングに依存して顕在化する実装バグ**
でした（本番でも、短時間にまとめてレコードを作る利用者で同じ取りこぼしが起き得た）。

### 決定的だった観察: 同一ファイルに「正しい実装」が既にあった

同じ repository ファイル内の他のページネーションクエリは、すべて一意列（主キー `ID`）を
tie-breaker に含んでいました。

```sql
-- 正しい keyset ページネーション（同ファイルの別メソッド）
WHERE OwnerID = @ownerID AND Status = @status AND
  (CreateTime < @createTime OR (CreateTime = @createTime AND ID <= @id))
ORDER BY CreateTime DESC, ID DESC
LIMIT @limit
```

問題の一覧 API が使うクエリ **だけ** がこの tie-breaker を欠いていました。
しかも tie-breaker 付きの正しいメソッドは既に実装・テストまで存在していたのに、
その一覧 API から配線されていませんでした（正しい部品はあったが使われていなかった）。

### 修正

一覧 API を tie-breaker 付きメソッドに切り替え、page token に一意列（`ID`）を
追加しました。cursor 側は同値を含めるため **`<=` を使う**のがポイントです。

```sql
(CreateTime < @createTime OR (CreateTime = @createTime AND ID <= @id))
ORDER BY CreateTime DESC, ID DESC
```

- 前ページ末尾の1件先（LIMIT+1 で先読みした行）の `(CreateTime, ID)` を cursor にする。
- 次ページは `CreateTime = cursor.CreateTime` の同値行のうち `ID <= cursor.ID` を
  **inclusive** に含めて再開するので、同値行を跨いでも取りこぼさない
  （`ID` は一意なので重複もしない）。
- 旧トークン（`CreateTime` のみ、`ID=""`）は `ID <= ""` が常に偽になり、従来どおりの
  strict `<` 挙動へフォールバックするため後方互換。

## 一般的な対策

### 1. ソートキーには必ず一意な tie-breaker を添える（推奨）

`ORDER BY <非一意列>` の keyset ページネーションでは、主キー等の一意列を
**第2ソートキー兼 cursor 成分**にする。

```sql
ORDER BY CreateTime DESC, ID DESC
WHERE (CreateTime < @ct OR (CreateTime = @ct AND ID <= @id))  -- 同値は <= で inclusive
```

cursor（page token）にも `(CreateTime, ID)` の **両方**を積む。片方だけだと再開点を
一意に特定できない。

### 2. cursor の不等号の向きに注意する

- 境界行を「次ページの先頭として再開する」設計なら、同値部分は **`<=`（inclusive）**。
- 「境界行はもう返し終えた」設計なら **`<`（exclusive）** だが、その場合でも同値行を
  跨ぐには第2キーの比較が必須。**strict `<` を単独の非一意列に使うのが最も危険**。

### 3. インデックスにも tie-breaker 列を含める

`(OwnerID, Status, CreateTime DESC)` ではなく
`(OwnerID, Status, CreateTime DESC, ID DESC)` のように、ソート＝スキャン順が
一意に定まるインデックスにする。エンジンによっては同値行の物理順が安定せず、
ページ間で順序が揺れる。

### 4. 時刻の同値衝突を前提に置く（丸めの犯人を DB と決めつけない）

「現実には同時刻にはならない」という思い込みは、並行書き込み下で簡単に破れる。
ソートキーに時刻を使うなら、同値は **必ず起きる** 前提で設計する。

衝突源は複数あり得るので切り分ける:

- **言語ランタイム/OS の壁時計分解能**。`time.Now()` は環境によって分解能が粗く、
  連続呼び出しで同一値を返す（本事例の主因。Docker Desktop on macOS 等で顕著）。
  ここが粗いと、DB がいくら高精度でも同値が入り込む。
- **コミットタイムスタンプや列の格納精度**（DB 側の丸め）。エンジンによって
  マイクロ秒/ナノ秒と異なる。**思い込みで決めつけず、対象エンジンで round-trip を
  実測する**（本事例では Spanner Omni / エミュレータともナノ秒保存で、DB は無罪だった）。
- **アプリが共有の固定時刻を全レコードに焼き込む**（テストの debug-time 等）。

### 5. 間欠失敗を安易に「フレーク」で片付けない

再実行で通る失敗でも、原因が並行タイミング依存の **実バグ** であることがある。
特に「セットアップ／全件走査で対象が見つからない」系は、データ取得ロジックの
取りこぼしを疑う。

## チェックリスト

以下が揃うと、このパターンのバグが潜在している可能性があります。

- [ ] keyset（カーソル）ページネーションを使っている
- [ ] `ORDER BY` のキーに **一意でない列**（`CreateTime` 等）が含まれる
- [ ] cursor 条件が **strict な不等号（`<` / `>`）のみ**で、一意列の tie-breaker が無い
- [ ] page token が非一意列の値だけを持ち、主キーを含めていない
- [ ] インデックスの末尾に一意列（PK）が無く、同値行のスキャン順が非決定的
- [ ] ソートキーが時刻で、その生成源（`time.Now()` 等の壁時計）が同一値を返し得る精度、
      かつ並行書き込みがある（丸めは DB とは限らない。まず時計源を疑い、round-trip を実測する）
- [ ] 同じコードベースの別メソッドには tie-breaker があるのに、この経路だけ欠けている

## 教訓

1. **非一意なソートキーの keyset ページネーションには必ず一意な tie-breaker を** —
   `ORDER BY t, id` と、cursor に `(t, id)` の両方を積むのが定石。
2. **同値は「起き得る」ではなく「必ず起きる」前提で** — 時刻の粒度丸めと並行書き込みで
   ソートキーは容易に衝突する。
3. **同一コードベースに正しい実装がないか確認する** — 正しい部品（tie-breaker 付き
   メソッド）が既にあるのに、一部の経路だけ古い/誤った実装を使っていることがある。
   これは「LIMIT + 後フィルタ」の事例と同じ学び。
4. **間欠的な E2E 失敗は、テストのフレークではなくプロダクションの実バグを検出している
   ことがある** — 「再実行で通る」で終わらせず、並行タイミング依存の取りこぼしを疑う。
