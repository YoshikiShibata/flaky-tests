# 壁時計の後方ステップ（NTP補正）による順序依存テストのフレーキー失敗

## TL;DR

- `ORDER BY <作成時刻> DESC` でソートする一覧APIのE2Eテストが、**隣接2件だけ綺麗に入れ替わって**落ちることがある。
- 真因はプロダクトバグでもテストロジックの誤りでもなく、**テスト実行中に macOS の `timed`（時刻同期デーモン）が NTP 補正でシステム壁時計を後方へステップさせた**こと。
- 作成時刻を `time.Now()` で設定している場合、壁時計が巻き戻ると「後に作成したレコードの方が作成時刻が小さい」逆転が起き、DBのソート結果がテストの「作成順」前提と食い違う。
- macOS なら **統合ログ（`log show`）の `timed` プロセスに `settimeofday` / `adjust,-N` の痕跡が残る**ので、後追いで確定できる。

---

## 1. 発生した問題（症状）

「特定の条件で絞り込んだレコード一覧を、作成時刻の降順で返す」一覧取得APIのE2Eテストで、
**隣接する2件のレコードがちょうど入れ替わった**形でアサーション失敗が出た。

```
index p   : got = レコードA, want = レコードB
index p+1 : got = レコードB, want = レコードA
```

入れ替わった2件は、識別子と一部の付随情報（割引の有無など）以外のフィールド
（所有者・ステータス・付与値など）は一致しており、**純粋に並び順だけ**が逆だった。

> 「隣接2要素の綺麗なスワップ」は、ソートの主キーが**同値（タイ）か逆転**したときの典型的なシグネチャ。
> まずここに着目すると原因の当たりが付けやすい。

---

## 2. 調査ステップ1：サーバ側は決定的か

並び順を疑う前に、サーバ側がそもそも非決定的でないかを確認する。

- 一覧クエリの `ORDER BY` を確認 → `作成時刻 DESC, 識別子 DESC` のように**完全に決定的**だった。
- レスポンス組み立ても、DBから取得したスライスの順序をそのまま保持して詰めるだけ。
  goroutine による並べ替えや map 経由の順序喪失もなし。

```sql
-- イメージ（一般化）
SELECT ... FROM <テーブル>
WHERE <絞り込み条件>
ORDER BY <作成時刻> DESC, <識別子> DESC
LIMIT @limit;
```

> **結論**: got（レスポンス）の順序は `作成時刻 DESC, 識別子 DESC` で決定的。
> `got = [A, B]` は `作成時刻(A) ≥ 作成時刻(B)` を意味する。

---

## 3. 調査ステップ2：期待値（want）の構築と矛盾

テストは、レコードを逐次作成しながら毎回 prepend して降順の期待配列を組んでいた。

```go
// 新しく作ったレコードを先頭に積む（=降順）
want = append([]*Record{newRecord}, want...)
```

作成時刻はレコード保存時に `time.Now()` で設定される（イメージ）:

```go
func setTimestamp(r *Record) {
    now := time.Now()
    if r.CreateTime.IsZero() {
        r.CreateTime = now
    }
    r.UpdateTime = now
}
```

作成ループは**完全に逐次**（各レコードの作成・確定処理はすべてブロッキング、goroutine なし）。
よって本来は次が成り立つはず:

```
逐次の実行順 == time.Now() の単調増加 == 作成時刻の昇順
⟹ want（実行順の逆＝最新が先頭） == got（作成時刻 DESC）
⟹ 一致するはずで、失敗しない
```

ところが実際は:

- `want = [B, A]`（B が先頭=index 小）
  ⟹ B の方が**後の反復で prepend された＝後に実行された**
- `got = [A, B]`
  ⟹ `作成時刻(A) ≥ 作成時刻(B)`

**つまり「後に実行した B の作成時刻の方が小さい」**。
逐次実行で `time.Now()` が単調増加なら絶対に起こらない。
→ **壁時計が単調増加していない**ことが論理的に確定する。

---

## 4. 調査ステップ3：実行ログのタイムスタンプ逆行

サーバの実行ログ（各行が unix エポックのタイムスタンプを持つ。作成時刻を刻むのと同じプロセスの時計）を
**出力順**に並べると、時刻が逆行していた。

```
（先に実行されたレコードの処理）  ts 1779677692700
                                ts 1779677693370
（次に実行されたレコードの作成）  ts 1779677691983   ← 約1.4秒 巻き戻り！
                                ts 1779677692068
                                ts 1779677693399   ← 元に戻る（前進）
```

- **実行順（=ログ出力順）**: A→B ではなく、先に処理されたのは「want で後ろに来るべき」レコード。`want = [B, A]` と整合。
- **作成時刻（壁時計）**: 先に実行した方が大きい値、後に実行した方が小さい値（巻き戻り後）。`got = [A, B]` と整合。

ここまでで「壁時計が後方へジャンプした」とほぼ断定できるが、OS のログで裏付けると確実。

---

## 5. macOS での「壁時計巻き戻し」ログ調査方法

macOS は統合ログ（unified logging）に時刻同期デーモン `timed` の挙動を残している。
`log show` で後追いできる。**バッファ保持期間内（ログ量により数時間〜数日）なら過去分も見える。**

### 手順 5-1. 問題のタイムスタンプをローカル時刻へ変換

ログのタイムスタンプ（ミリ秒なら秒に直す）を `date -r` で変換し、調査窓を決める。

```bash
# 例: 1779677693370(ms) → 1779677693(s)
date -r 1779677693   # => 2026年 5月25日 11時54分53秒 JST
date -r 1779677691   # => 2026年 5月25日 11時54分51秒 JST
```

→ 巻き戻りは **2026-05-25 11:54:51〜53 JST** 付近、と窓が絞れる。

### 手順 5-2. `timed` のログを窓で抽出

```bash
log show --start "2026-05-25 11:53:00" --end "2026-05-25 11:56:30" \
  --predicate 'process == "timed"' --info --debug \
  | grep -Ei 'settimeofday|Setting system time|adjust,-|RejectATime|Received time'
```

> 補足:
> - `--info --debug` を付けないと `timed` の詳細ログは出ない。
> - zsh ではコマンドを1行で書く（複数行＋バックスラッシュ継続でクォートが壊れ `too many arguments` になることがある）。
>   `log` が別物に解決される環境では `/usr/bin/log` を明示する。
> - 過去ログが流れていて見えない場合はリアルタイム監視に切り替える:
>   `log stream --predicate 'process == "timed"' --info --debug`

### 手順 5-3. 出力の読み方（実際に取れた証拠）

```
11:54:51.576109  timed: [com.apple.settimeofday:] settimeofday({0x6a13b9fb,0x8b87b}) == 0
11:54:51.576208  timed: Setting system time to Mon May 25 11:54:51 2026 from TMTimeSynthesizer
11:54:51.603870  timed: cmd,apply,src,settimeofday,…,adjust,-2.044957995,success,1
11:54:53.613854  timed: TDTF,RejectATime,…,innv,-2.044978,…,source,NTP
11:54:53.612950  timed: Received time Mon May 25 11:54:51 2026±0.01 from "NTP"
```

| ログ | 意味 |
|---|---|
| `settimeofday({0x6a13b9fb,0x8b87b})` | **緩やかな slew ではなく時刻を一気に書き換えるハードなステップ**。`0x6a13b9fb` = `1779677691`（tv_sec）、`0x8b87b` = `571003`（tv_usec）→ 時計を `1779677691.571` に設定し直した |
| `adjust,-2.044957995` | この適用で壁時計が **約 −2.04 秒 後方へジャンプ**した |
| `innv,-2.044978` (innovation) | ローカル時計が NTP 比で約 +2.04 秒**進んでいた**と判定 → それを打ち消すため後方補正 |
| `Received time … from "NTP"` | NTP サーバから受信した正しい時刻 |

> macOS は小さなズレは slew（周波数を微調整して徐々に追従）で吸収するが、
> **閾値（おおむね数百ms〜秒オーダー）を超えると `settimeofday` で一気にステップ**する。
> 今回は約2秒ズレていたためステップになった。

### 手順 5-4.（任意）スリープ/復帰の確認

スリープ復帰でも壁時計がジャンプすることがある。電源管理ログで確認:

```bash
pmset -g log | grep -Ei 'Wake|Sleep|DarkWake' | tail -50
```

今回はスリープではなく NTP ステップが主因だった。

### 手順 5-5. step と slew の違い（なぜ settimeofday だけ見ればよいか）

時刻同期の補正には **step** と **slew** の2種類があり、調査では **step（`settimeofday`）だけを見れば十分**。
理由は「`time.Now()` の**値**を変えるのは両方だが、**値を後退（非単調に）させうるのは step だけ**」だから。

| 操作 | ログ上 | 時計への作用 | `time.Now()` の値 | 単調性 |
|---|---|---|---|---|
| **step** | `src,settimeofday` | 時刻を**不連続に書き換え**（瞬間ジャンプ） | 変わる（前後どちらにも飛ぶ） | **後退しうる** ← 順序破壊の原因 |
| **slew** | `src,adjtime`（`ntp_adjtime`） | tick の**速度を一時的に増減**して徐々に収束 | 変わる（レートが変化） | **後退しない**（遅らせるだけ・止まりもしない） |

ポイント:

- slew は「進みすぎ」を直すときも、時計を巻き戻すのではなく **“ゆっくり進ませる”** ことで補正する
  （macOS/BSD はレート変更に上限があり、概ね数百ppm）。よって **`time.Now()` が前回より小さい値を返すことは原理的にない**。
- したがって `adjust` が負の slew（＝遅らせる方向の補正）が何件あっても、**順序（作成時刻の並び）は壊れない**。
  順序を壊しうるのは**後退 step だけ**。
- Go の `time.Now()` は壁時計（CLOCK_REALTIME）に加えモノトニッククロック（CLOCK_MONOTONIC）も同時に読むが、
  後者は step/slew どちらの影響も受けない。ただし DB に保存する作成時刻や、別々の `time.Now()` 同士で
  **絶対時刻を比較**する場面ではモノトニック成分は剥がれ、**壁時計の値**が使われる。だから今回は step の後退が効いた。

> **実測（このマシン・保持ログ約20.7時間）**: slew は56件（うち遅らせる方向＝負が30件、最大でも −38ms）あったが、
> これらは一切時計を後退させていない。実際に時計が後退したのは step（`settimeofday`）の4件のみ
> （§8 参照。実害は −2.04 秒の1件）。

## 6. なぜこの種のテストは脆いのか

このパターンのテストは次を暗黙の前提にしている:

```
逐次の作成順 == time.Now() の単調増加 == DB の作成時刻順
```

しかし DB 仕様は `ORDER BY 作成時刻 DESC, 識別子 DESC`。次の2つに対して脆い:

1. **壁時計の非単調性（今回）** — NTP ステップやスリープ復帰で後方ジャンプすると作成時刻が逆転する。
2. **作成時刻のタイ** — 2件が同一タイムスタンプになると DB は第2キー（識別子）で tie-break するが、
   テストはそれを再現していないので作成順と食い違う。

### 後退ステップが実害になる条件（危険窓 ≈ δ）

後退ステップ1回が実害を生むには、次の3つが**同時に**必要:

1. 時計が後退する（step が起きる）
2. **順序が意味を持つデータ**が、その後退をまたいで作られる
3. それが**タイムスタンプ順で比較**される（DB の `ORDER BY 時刻` など）

後退量 δ の step は、**直前 δ 秒ぶんの時刻を“もう一度やり直す”**ことになる。
つまり「step 直前 δ 秒に作ったデータ」と「step 直後 δ 秒に作ったデータ」のタイムスタンプが重なり/逆転する。
→ **危険窓 ≈ δ**。順序を狂わせられるのは **δ より短い間隔で作られた2件**だけ。

当たる確率は「**後退量 δ ÷ データ作成間隔**」でおおよそ決まる:

- `δ < 作成間隔` → 窓が1件ぶんに満たず **まず当たらない**
- `δ ≥ 作成間隔` → 隣接ペアを逆転させうる **危険**

このマシンの実測（作成間隔 約0.6秒）での見積もり:

| 後退 step | δ（後退量） | 危険窓 | 判定 |
|---|---|---|---|
| 01:29 | 0.053 秒 | 〜53ms | 1件ぶん未満 → まず当たらない |
| 04:40 | 0.128 秒 | 〜128ms | 同上 → ほぼ無害 |
| 09:38 | 0.120 秒 | 〜120ms | 同上 → ほぼ無害 |
| **11:54** | **2.045 秒** | **〜2秒** | **約3件ぶんをまたぐ → 逆転（HIT）** |

> つまり「後退が4回あった＝失敗チャンス4回」ではない。実害を出せたのは δ が作成間隔より十分大きい
> **−2.04秒の1件だけ**。逆に、**データ作成がミリ秒間隔のような高速ケース**では、数十msの小さな後退でも
> 逆転を起こしうる（`δ ≧ 作成間隔` になるため）。

### 本番（Cloud Spanner）では基本的に起きない

この問題が現実的に起きるのは **ローカル開発機**。本番では以下の理由でまず顕在化しない:

- **作成時刻を Spanner のコミットタイムスタンプ（`PENDING_COMMIT_TIMESTAMP` / `spanner.CommitTimestamp`）で
  打つ設計**なら、時刻は Spanner サーバ側で **TrueTime に基づきコミット順に単調増加**する形で割り当てられ、
  アプリ側のローカル壁時計に一切依存しない。後退ステップの影響を受けない。
  - 一方、**作成時刻をアプリの `time.Now()` で設定している場合**（今回のケースもこれ）は、本番でも
    「アプリ稼働ホストの壁時計」に依存する点に注意。ただし次の理由で実害になりにくい。
- **GCP のホストクロックは規律されている** — Google の時刻同期はうるう秒スミアリング等で連続化されており、
  ローカル Mac で見られたような秒オーダーの後退ステップを避ける設計。後退ステップ自体がまず起きない。
- そもそも本番は**多数リクエストが並行**してデータを作るため、「逐次作成順 == 時刻順」という前提が成立しない。
  まともな設計は**順序を厳密に壁時計へ依存させない**（識別子で tie-break、ページトークンに `(時刻, 識別子)` を持つ等）
  ことが前提になっている。

> 要するに今回のフレーキーは、「**ローカル開発機の壁時計が後退しうる**」という環境特有の事情と、
> 「テストが壁時計の単調性を暗黙に仮定している」という前提の**組み合わせ**で起きるもの。
> 本番のデータ整合性そのものの問題ではない。

### 「本物の Spanner を使えば起きない」か？ → 時刻の発番元しだい

「ローカルの Spanner エミュレータではなく本物の Spanner を使えば回避できるのでは」と考えがちだが、
**これは条件付きで、今回のケースでは当てはまらない**。鍵は **作成時刻を誰が発番しているか**。

今回のコードは作成時刻を**アプリ側の `time.Now()`**（ローカルで動くサービスプロセス）で打ち、
その値をミューテーションに載せて Spanner へ渡している。Spanner（エミュレータでも本物でも）は
**受け取った値をそのまま保存するだけ**で、時刻の出どころは Spanner ではなく**ローカルアプリの壁時計**。

| 作成時刻の発番元 | ローカルのエミュレータ | 本物の Spanner |
|---|---|---|
| **アプリの `time.Now()`**（← 今回） | ローカル壁時計依存 → 後退の影響を受ける | **同じくアプリの壁時計依存 → 本物でも受ける** |
| **Spanner コミットタイムスタンプ**（`PENDING_COMMIT_TIMESTAMP`） | エミュレータは**ローカル host clock** で発番 → **受ける** | サーバ側 **TrueTime** で発番 → **受けない（安全）** |

- **今回のケース（`time.Now()` 発番）では、本物の Spanner に替えても直らない。** 時刻はローカルアプリの
  壁時計から来ており、Spanner はその器でしかない。**エミュレータが犯人なのではなく、ローカルアプリの
  壁時計の後退が犯人**。
- 「本物の Spanner なら起きない」が成り立つのは、**コミットタイムスタンプで発番する設計**にした場合のみ
  （右下のセル）。本物の Spanner はサーバ側 TrueTime で単調な時刻を割り当てるので後退の影響を受けない。
- ただし**その設計でもエミュレータは安全ではない**。エミュレータは TrueTime を持たず、コミットタイムスタンプも
  **ローカルの host clock** で生成するため、ローカル時計が後退すれば同じ問題が起きうる。

> まとめ: 「本物の Spanner で回避できる」のは **`コミットタイムスタンプ発番` × `本物の Spanner`** の
> 組み合わせに限られる。アプリの `time.Now()` で発番している限りは、エミュレータでも本物でも
> （＝ローカル実行で壁時計が後退する限り）起きうる。

### Cloud Run（本番の実行環境）ではどうか

本番はサービスを Cloud Run 上で動かしている。結論: **今回と同種の「秒オーダーの後退ステップ」はまず起きない**。
ただし「タイムスタンプ順 == 作成順」を 100% 保証するものではない。

- **後退ステップは起きにくい** — Cloud Run の実行基盤（Google 管理ホスト）のクロックは Google の時刻同期で
  規律されており、うるう秒スミアリング等で**連続・単調**に保たれ、ローカル Mac で見た秒オーダーの後退ステップを
  避ける設計。`time.Now()` が後退するという今回の犯人そのものが本番ではまず発生しない。
  → **このフレーキーは本番では再現しないと考えてよい**。
- **ただしインスタンス間スキューは残る** — Cloud Run はオートスケールで**複数インスタンス**が並行稼働する。
  各インスタンスのクロックが個別に単調でも、**インスタンス間の時刻は完全一致しない**（Google の同期は概ね
  ms 以下に収まるが 0 ではない）。レコード A をインスタンス1、B をインスタンス2 が**ごく近接した時刻**に作ると、
  `time.Now()` の値が実時間の前後関係を反映しないことがある。これは後退ステップとは**別機構**で、ズレは
  **ms オーダー**と桁が小さく、多くの場合は第2キー（識別子）の決定的な tie-break で吸収される。
- **厳密な単調順序が要るなら** — インスタンス数や時計に依存せず保証したいなら、アプリの `time.Now()` ではなく
  **Spanner のコミットタイムスタンプ（TrueTime）** を使うのが正攻法。

## 7. 今回の対応方針

- **対策は見送り**。原因が壁時計の非単調性という環境要因であり、作成時刻の降順に依存する他のE2Eテストも
  同様に揺れうるため、個別テストを直しても根本解決にならず、横展開コストに見合わないと判断。
- 同種の「順序系E2Eの隣接スワップ失敗」を見たら、まず環境要因（クロックステップ）を疑い、
  実行ログのタイムスタンプ逆行と `log show`（timed）で確認のうえ**再実行**する。

### （将来 恒久対策する場合の選択肢）

- want / got を**識別子をキーに集合比較**する。
- 両者を**同一のソートキー（作成時刻, 第2キー降順）でソートしてから**比較する
  （タイには効くが壁時計逆行には無力）。
- テスト内で作成時刻を制御・注入できるようにする。

## 8. 発生頻度・なぜ稀か

この事象は**極めて稀**である。ローカルでのE2Eテスト実行を 2018 年から続けてきて、
今回（2026年）が**初遭遇**だった。成立には複数の低確率条件が**同時に揃う**必要がある:

1. **NTP が slew ではなく step した** — OS は普段ズレを slew（周波数の微調整で徐々に追従）で吸収し、
   `settimeofday` による一気の後方ステップは**オフセットが閾値（秒オーダー）を超えたときだけ**起こる。
   定常的に時刻同期できているマシンでは、そもそもローカル時計が数秒ずれること自体が珍しい
   （スリープ復帰・ネットワーク再接続・オフライン後の復帰・VM/コンテナの時刻ドリフト・
   高負荷で時刻同期デーモンが遅延、等が引き金になりうる）。
2. **その後方ステップが、順序を検証している2レコードの逐次作成の“隙間”にちょうど着弾した** —
   窓は数百ミリ秒〜秒オーダーと狭い。
3. **テストが `time.Now()` 由来の作成時刻順に依存**していて、第2キーの tie-break を再現していない。
4. **セットアップが長時間**（多数レコードを十数分かけて作成）→ ステップが実行中に当たる露出時間が長い。

確率はこれらの**積**なので、「長年で初めて」はむしろ自然。
裏を返すと、長時間ランニングテストだからこそ拾えた稀少事象とも言える。

> なお、近年マシン／OSバージョン／ネットワーク環境が変わって**ステップが起きやすくなった**可能性もある。
> 再発が続くようなら `log show --predicate 'process == "timed"' --info --debug | grep adjust` で
> ステップ頻度の傾向を確認するとよい。

### このマシンでの実測（2026-05-25）

問題判明後、保持ログ範囲で `settimeofday`（＝slew ではなくハードなステップ）の apply を全件抽出した。
ただし `timed` の統合ログはこのマシンでは**約半日でローテーション**され、本日分（00:38〜12:26、約12時間）しか残っていなかった。
その範囲で計10件（前進6・後退4）。**後退（巻き戻し）は4件**:

| 時刻 (JST) | adjust | 備考 |
|---|---|---|
| 01:29:13 | −0.053 秒 | 微小 |
| 04:40:50 | −0.128 秒 | 微小 |
| 09:38:43 | −0.120 秒 | 微小 |
| **11:54:51** | **−2.045 秒** | **← 今回テストを壊した巻き戻し** |

ポイント:

- 後退4件中3件は数十〜100ms程度の微小補正で、順序に影響する可能性はほぼない。
  **実害があったのは 11:54:51 の −2.04 秒だけ**。「複数条件の積」という稀少性が実データでも裏付けられた。
- 直後の **12:26:55 に +2.05 秒の前進ステップ**があり、−2.04秒をほぼ打ち消していた。
  つまり昼前後に**約2秒の時刻オシレーション**が起きており、通常の NTP 微補正（〜数百ms）と桁が違う。
  時刻源の不安定さ・ネットワーク経路・RTC/発振器まわりが疑わしく、**再発リスクの兆候**かもしれない。

### 長期監視（統合ログは短命なので追記保存する）

傾向を継続的に追うなら、ステップ発生時にファイルへ追記しておくと確実:

```bash
# 後退（巻き戻し）のみ追記
log stream --predicate 'process == "timed" AND eventMessage CONTAINS "cmd,apply,src,settimeofday"' --info --debug \
  | grep --line-buffered 'adjust,-' >> ~/clock_backward_steps.log
```

`grep 'adjust,-'` を外せば前進ステップも記録できる。

## 9. 教訓・チェックリスト

- [ ] 「隣接2要素の綺麗なスワップ」失敗を見たら、ソート主キーの**タイ or 逆転**を疑う。
- [ ] サーバ側の並び順が決定的か（クエリの `ORDER BY` と組み立てロジック）をまず確認する。
- [ ] `time.Now()` 由来のタイムスタンプに順序を依存させていないか確認する。
      **壁時計は単調増加とは限らない**（NTP ステップ・スリープ復帰）。
- [ ] macOS なら `log show --predicate 'process == "timed"' --info --debug` で
      `settimeofday` / `adjust,-N` を探せば、巻き戻しを後追いで確定できる。
- [ ] `date -r <epoch秒>` で問題タイムスタンプをローカル時刻に変換し、ログの調査窓を絞る。
