# 業務時間境界をまたぐテストデータによるフレーキーテスト

## パターン名

**テストデータの相対時刻がドメイン固有の日付境界をまたぐ**

## 問題の本質

テストが `time.Now()` を基準に過去の相対時刻（例: `-35分`）でデータを作成する場合、
ドメイン固有の日付境界をまたぐと「今日」のデータが「昨日」として扱われ、テストが失敗する。

日付境界は必ずしも `00:00` ではない。業務ドメインによって異なる境界時刻が使われる:
- 朝4時（ゲーム系サービスの1日の切り替え時刻）
- 営業開始時刻（会計日の区切り）
- シフト切り替え時刻

## 発生条件

以下の3条件がすべて揃った場合に発生する:

1. **テストが `time.Now()` を基準時刻として使用**
2. **テストデータが基準時刻から相対的に過去の時刻で作成される**（例: `baseTime.Add(-35 * time.Minute)`）
3. **テスト実行時刻がドメイン固有の日付境界の直後**（例: 境界が4:00の場合、4:00〜4:35）

## 実例

テストは `baseTime := time.Now()` を基準に、複数のイベントを異なる時刻で作成:

```
baseTime           : イベントA（最新）
baseTime - 10分    : イベントB
baseTime - 35分    : イベントC（最古）
baseTime - 24時間  : イベントD（昨日として除外される想定）
```

ドメインの日付境界は朝4時（JST）。

**正常時（例: baseTime = 10:00）:**
```
日付境界:    04:00
イベントC:   09:25  ← 今日の範囲内 OK
→ 今日のイベント数 = 3（A + B + C）
```

**異常時（例: baseTime = 4:20）:**
```
日付境界:    04:00
イベントC:   03:45  ← 前日の範囲に入る NG
→ 今日のイベント数 = 2（C が除外される）
```

## 対策パターン

### baseTime を境界から十分に離す

```go
// Before（脆弱）
baseTime := time.Now()

// After（堅牢）
// 最も過去のデータは baseTime - 35分 に作成されるため、
// 日付境界（朝4時）をまたがないよう baseTime が 4:35 以降であることを保証する
baseTime := time.Now()
today0436 := time.Date(baseTime.Year(), baseTime.Month(), baseTime.Day(), 4, 36, 0, 0, baseTime.Location())
if baseTime.Before(today0436) {
    baseTime = today0436
}
```

### ポイント

- **必要な最小オフセットを計算する**: 最も過去のデータの相対時刻（この例では -35分）に余裕を加えた時刻（4:36）を最小値とする
- **未来の固定時刻ではなく条件付きで調整する**: 通常時は `time.Now()` をそのまま使い、境界付近の場合のみ調整する
- **コメントで理由を明記する**: なぜその時刻を選んだかを後の読み手のために記載する

## 一般化した教訓

1. **テストデータの相対時刻の最大スパンを把握する**: テスト内で最も過去のデータが何分前に作成されるかを確認し、その分だけ境界から離す
2. **ドメイン固有の日付境界を把握する**: `00:00` だけでなく、ビジネスロジック固有の境界時刻を意識する
3. **CI の実行時刻は制御できない**: 任意の時刻に実行されても安全なテストを書く
4. **「毎日特定の時間帯だけ失敗する」テストはこのパターンを疑う**: 失敗する時間帯とドメインの日付境界を照合する
