# ユニークキー INSERT でも「最新を選ぶグローバルセレクタ」を奪うと writer になる

## 概要

自分専用のユニークキー行しか INSERT/DELETE しないテストでも、その行が「最新版を選ぶグローバルなセレクタ（`ORDER BY version DESC LIMIT 1` 等）」の選択結果を差し替えてしまう場合、そのテストは実質的に**グローバル状態の writer** である。にもかかわらず「ユニークキーしか触らないから」と Shared（共有）ロックで運用すると、同じグローバル状態を Shared で読む並行テストとの間で flaky を生む。

「ユニークキー INSERT のみ → Shared で安全」という一般則（→ [全データ更新ジョブによるテスト間干渉](./global-data-modification-interference.md)）の**重要な例外**。

## 症状

- 対象テストを単独実行すると常に成功する（並行する writer が居ないため）。
- 全 E2E 並列実行で、ランキング／最新スナップショット／「現在有効な設定」などを読むテストがランダムに失敗する。
- 失敗テストは「同じリストを 2 回取得して整合を見る」「取得結果に対して付随情報の有無を確認する」構造を持ち、1 回目と 2 回目で**取得対象集合そのものが入れ替わっている**。
- データが無い実行では早期 return して trivial PASS するなど、**実行ごとに前提データの有無でも揺れる**。

## 根本原因

### 問題のある構成

- **writer テスト W**: `version = now()` 等で採番した自分専用の新バージョンを INSERT する。このバージョンは採番規則上ほぼ常に「最大」になるため、グローバルな「最新セレクタ」が W のデータを指すようになる。テスト終了時に `t.Cleanup` で自分の行を DELETE する。W はユニークキーしか触らないので Shared ロックで運用されている。
- **reader テスト R**: 同じグローバル状態を Shared ロックで読む。「最新」を 2 回引き、その間に取得集合が安定であることを暗黙に前提とする。

### 発生メカニズム

1. W（Shared）が新バージョンを INSERT → グローバル最新セレクタが W のデータを指す。
2. R（Shared）が並行実行され、1 回目の読み取りで W のデータ集合を観測。R はその集合に対して副作用（例: 全件にフラグ付け）を行う。
3. W が完了し、`t.Cleanup` で自分の行を DELETE。
4. R の 2 回目の読み取りでは、下に埋もれていた**別バージョン**が最新として復活。R が 1 回目に対して行った副作用は新集合に一致せず、検証が失敗。

Shared 同士は並列実行できてしまうため、W と R が重なる。「W はユニークキーしか INSERT しない」は真だが、**INSERT がグローバルな派生状態（最新ポインタ）を変える**点を見落としている。

## 一般化：これに該当する「隠れ writer」

自分の行しか触らなくても、次のようなグローバル派生状態を動かすなら writer であり Exclusive が必要：

- `ORDER BY created_at / version / snapshot_id DESC LIMIT 1` で引かれる「最新」
- `MAX(version)` / `SELECT ... WHERE version = (SELECT MAX(...))` で引かれる「現行」
- 「期間が現在を含む唯一のレコード」「`is_active = true` の唯一行」など、グローバルに一意な選択結果
- ランキング・スナップショット・フィーチャーフラグの現行値・有効な料金表など

判定基準: **「自分が INSERT/DELETE した行が、他テストの読み取りクエリの選択結果（どの行が返るか）を変え得るか？」** が Yes なら writer。

## 解決策

- グローバル最新セレクタを差し替える INSERT/DELETE を行うテストは **Exclusive** ロックを取得する。これにより同じ状態を Shared で読む並行テストと排他され、読み取り中の差し替えが防止される。
- ロックの由来は「他テストが取っているから」ではなく「**この書き込みがグローバルな選択結果を変えるから**」で判断・明記する。
- reader 側（Shared）は writer が Exclusive である限り正しい。reader を頑健化（2 回読みの前提を捨てる等）するより、**writer を正しく Exclusive にする**方が根治。

### Shared を使ってよい条件（更新版）

[global-data-modification-interference.md](./global-data-modification-interference.md) の条件に 1 項追加：

1. テスト内で作成したユニークキー行のみを INSERT する
2. 既存行を更新・削除しない
3. 他テストが作成したデータに依存しない
4. **その INSERT/DELETE が、他テストの読み取りクエリの「どの行が返るか」を変えない**（最新セレクタ・MAX・唯一の有効行などを動かさない）

4 を満たさないなら、ユニークキーのみでも Exclusive。

## 検出の手がかり

- 失敗ログで「取得件数が 1 回目と 2 回目で違う」（例: 50 件 → 6 件）。集合の入れ替わりを示す決定的証拠。
- ロックログで、失敗 reader の Shared 保持中に同じリソースの Exclusive が一切取得されていないのに状態が変わっている → **Shared writer の並行**を疑う。
- 同リソースへの書き込みテストを全列挙し、ロックモードを突き合わせる。1 つだけ Shared なら、それが犯人である可能性が高い。

## 教訓

1. 「ユニークキー INSERT だから Shared で安全」は、**グローバルな最新/現行セレクタを動かさない**場合に限る。
2. INSERT そのものより、**その INSERT が他テストの読み取り結果集合を変えるか**で writer/reader を判定する。
3. ロック機構を移行・リファクタする際（例: 独自ロック→共通 MultiLock）は、既存のロックモードを**温存せず**、各テストの writer/reader 性質を再点検する。誤ったモードは「flaky 対策」の名目で安易に入り、並列化を経て起爆し、機構移行で温存されがち。
4. 「flaky 対策コミット」がさらに別の flaky を仕込むことがある。対策時は対象テスト単体でなく、**同リソースに触れる全テストのロック整合**を見る。
