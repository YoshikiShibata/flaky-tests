# 並行取得の「先着エラー」非決定性とエラー→ステータスマッピングの取りこぼし

## 概要

`errgroup` で複数の取得処理を並行実行し、いずれか一つでもエラーが起きたら全体を失敗させる構造において、**同一の論理的失敗条件**（例: 「エンティティが存在しない」）が**複数の異なるセンチネルエラー**として現れることがある。

`errgroup.Wait()` は**最初に発生したエラーだけ**を返すため、どのセンチネルが返るかは goroutine のスケジューリング次第で**非決定的**になる。

このとき下流の「エラー → ステータスコード」マッピング層が、起こりうるセンチネルのうち**一部しかマッピングしていない**と、外部から観測されるステータスコードが実行ごとに変わる。これを検証するテストは Flaky になる。

重要なのは、これは**テストの問題ではなくプロダクトの非決定性のバグ**だという点。同一リクエストがタイミング次第で異なるステータスコード（例: `NotFound` と `Internal`）を返す。テストはその実在のバグを露出させただけ。

## 発生条件

1. ハンドラ／ユースケースが、ある集約の複数の関連データを `errgroup` で**並行取得**している
2. 集約が存在しない（あるいは前提条件を満たさない）とき、**複数の取得処理が同時にエラーになる**
3. それらのエラーは**意味的には同じ条件**を表すが、**別々のセンチネル**（`ErrXNotFound`, `ErrYNotFound` …）として定義されている
4. 下流のマッピング層（gRPC status / HTTP code / エラーレスポンス変換）が、それらのセンチネルの**一部しか目的のコードにマッピングしていない**（残りは `default` で `Internal` 等に落ちる）
5. テストがその外部ステータスコードを assert している

## なぜ発見が遅れるか

- `errgroup.Wait()` は「最初に失敗した goroutine」のエラーを返す。どれが最初に失敗するかは各取得処理の**所要時間の差**に強く依存する。
- 単体実行（対象テスト1本だけ）では goroutine 競合がほとんど無く、**速い取得処理が常に先に完了・先に失敗**する。
  - 例: 主キー1行 read は、カラム検索／スキャン／JOIN を伴う取得より速く、ほぼ確実にレースに勝つ。
- そのため、速い側のセンチネルがマッピング済みであれば、**単体・少並列では延々と成功**し、繰り返し実行しても再現しない。
- CPU 競合が激しい一括並列実行下でのみ、速いはずの goroutine がデスケジュールされ、**遅い側のセンチネルが勝つ**ことがある。マッピング漏れのセンチネルが勝ったときだけ失敗する。
- 結果として「ローカルの一括実行や CI でたまに落ちるが、その1本だけ流すと必ず通る」という典型的な Flaky の様相を呈する。

## 具体例

### 状況

あるリソースのサマリを返す RPC `GetX` がある。仕様（proto コメント等）では「リソースが存在しない場合は `NotFound`」と定められている。

ユースケースは、同じユーザーに紐づく複数の関連行を `errgroup` で並行取得する:

```go
eg, ctx := errgroup.WithContext(ctx)

eg.Go(func() (err error) {
    a, err = repo.FindA(ctx, userID) // 未登録なら ErrANotFound（主キー read・速い）
    if err != nil {
        return liberrors.Wrap(err, "failed to FindA")
    }
    return nil
})

eg.Go(func() (err error) {
    b, err = repo.FindB(ctx, userID) // 未登録なら ErrBNotFound（カラム検索・遅い）
    if err != nil {
        return liberrors.Wrap(err, "failed to FindB")
    }
    return nil
})
// ... 他にもいくつかの並行取得 ...

if err := eg.Wait(); err != nil {
    return nil, err // 「最初に失敗した1つ」だけが返る
}
```

ハンドラのマッピングは片方しか拾っていない:

```go
if err != nil {
    switch {
    case errors.Is(err, ErrANotFound):
        return nil, status.Code(codes.NotFound, err)
    default:
        return nil, status.Code(codes.Internal, err) // ErrBNotFound はここに落ちる
    }
}
```

### 失敗の現れ方

未登録ユーザーで `GetX` を呼ぶと `FindA` と `FindB` が両方失敗する。

- `FindA`（`ErrANotFound`）が勝つ → `NotFound` → テスト成功
- `FindB`（`ErrBNotFound`）が勝つ → `default` → `Internal` → テスト失敗

```
get_x_test.go:NN: status.Code(err) is Internal, want NotFound
```

同じテスト・同じ入力なのに、勝つ goroutine が変わるだけで結果が変わる。

### 解決策（推奨: マッピングを網羅的にする）

同じ論理条件を表すセンチネルは、**すべて同じ結果にマッピング**する:

```go
switch {
case errors.Is(err, ErrANotFound), errors.Is(err, ErrBNotFound):
    return nil, status.Code(codes.NotFound, err)
default:
    return nil, status.Code(codes.Internal, err)
}
```

これで**どの goroutine がレースに勝っても**確定的に `NotFound` を返す。同種の他ハンドラが既に両方をマッピングしていれば、それが規約上の正解であることの裏付けになる。

### 別の解決アプローチ

- **発生源で正規化する**: ユースケース層で「集約が存在しない」を表す関連センチネルを一つの代表エラーに畳んでから返し、マッピング層の分岐を単純化する。
- **意味的にまとめる**: そもそも複数の「○○ NotFound」を個別センチネルに分ける必要があるか見直す。外部契約として区別不要なら統合する。
- 並行をやめて逐次化するのは、非決定性は消えるが「先に評価した側のエラーだけ返る」点は残り、レイテンシも犠牲になるため通常は非推奨。**マッピングの網羅**が本筋。

## レースで負ける分岐を確定的に再現する検証テクニック

「速い側が常に勝つ」ために再現できない分岐は、**速い goroutine に一時的な遅延を注入**して遅い側を確定的に勝たせると、修正後の挙動を実証できる:

```go
eg.Go(func() (err error) {
    time.Sleep(100 * time.Millisecond) // TEMP: 遅い側を勝たせて検証。確認後に必ず revert
    a, err = repo.FindA(ctx, userID)
    ...
})
```

- 注入後にテストが目的のコード（例 `NotFound`）を返して通れば、マッピング修正が両分岐で正しいことの証明になる。
- 検証用の遅延は**必ず revert** する。コミットに残さない。
- `errgroup.WithContext` は最初のエラーで他 goroutine の `ctx` をキャンセルするが、`Wait()` が返すのは「最初のエラー」で固定されるため、遅延注入後の遅い goroutine が後から別エラー（`context.Canceled` 等）を返しても結果は変わらない。

## 一般化したパターン

| 項目 | 内容 |
|------|------|
| 根本原因 | 同一論理条件が複数センチネルで現れ、`errgroup.Wait()` の「先着エラー」が非決定的、かつマッピング層がセンチネルを取りこぼす |
| 性質 | テストの不備ではなく**プロダクトの非決定的挙動**（同一リクエストが異なるステータスを返す）の露出 |
| 混入契機 | 並行取得の追加／センチネルエラーの新設／関連行が増えたとき。マッピング層の更新漏れ |
| 発見の遅れ | 速い取得処理がほぼ常にレースに勝つため、単体・少並列では再現しない。高並列・高負荷時のみ遅い側が勝って顕在化 |
| 検出方法 | エラーを返しうる全 goroutine のセンチネルを列挙し、マッピング層の分岐が**全部を網羅**しているか確認。proto/API 仕様のエラー定義と突き合わせる |
| 解決方法 | 同じ論理条件のセンチネルをすべて同じ結果にマッピング（または発生源で正規化）。網羅性を担保 |
| 検証方法 | 速い goroutine に一時遅延を注入して負ける分岐を確定再現 → 確認後 revert |

## レビュー・実装チェックリスト

`errgroup`（あるいは並行取得 + 単一エラー返却）と、その下流のエラー→ステータスマッピングを書く／変更するときに確認する:

1. **その errgroup 内でエラーを返しうる goroutine は全部で何本か？** それぞれどのセンチネルを返しうるか列挙したか
2. **意味的に同じ条件**（「存在しない」「権限がない」等）を表すセンチネルが複数ないか。あるなら**すべて同じステータス**にマッピングしているか
3. マッピングの `default`（`Internal` 等）に、本来は別コードであるべきセンチネルが**紛れ込まないか**
4. 同種の既存ハンドラと**マッピングが一致**しているか（片方だけ網羅、はバグの兆候）
5. API 仕様（proto コメント等）が定めるエラーコードと、実装のマッピングが**一致**しているか
6. 新しい関連取得や新しいセンチネルを追加したとき、**マッピング層の更新を忘れていないか**

## 関連概念

- **先着エラー（first-error）セマンティクス**: `errgroup.Group` は最初に non-nil を返した goroutine のエラーのみを保持（`sync.Once`）し、`Wait()` で返す。後続の goroutine がどんなエラーを返しても上書きされない。
- **エラーの正規化 vs 透過**: ドメイン固有の詳細センチネルを下流にそのまま流す（透過）と、マッピング層が全センチネルを知る責務を負う。境界で代表エラーに正規化すると分岐が単純になる。
- **テストは実バグの露出器**: Flaky の修正でテスト側を緩めると、同一リクエストが時々 `Internal` を返す実害（クライアントのリトライ挙動・監視アラート）を見逃す。直すべきは実装側。
