# 共有フィクスチャプールの再利用による残留状態の干渉

## 概要

テスト間で共有・再利用されるフィクスチャプール（ユーザー / アカウント等のエンティティを使い回す仕組み）から取得したエンティティに、過去のテストで蓄積された状態（過去の操作履歴・ステータス遷移・関連レコード等）が残り、「初期状態 / クリーンなエンティティ」を前提とするテストが flaky に失敗する。

## 症状

- 単独実行では常に成功するが、全テスト並列実行でランダムに失敗する。
- 「エラーになるはず」の検証で逆にエラーが返らず成功してしまう（あるいはその逆）。失敗が「期待した分岐に入っていない」形で現れる。
- 失敗時のエンティティが、当該テストがセットアップしていない状態（操作履歴あり・ステータス変更済み・関連レコードあり等）を持っている。

## 根本原因

### プールの再利用メカニズム

- フィクスチャプールは、テスト終了時（cleanup）にエンティティを「未使用」フラグに戻すだけで、**同一の実体（同じ ID）を次のテストに再配布**する。
- 過去のテストがそのエンティティに対して行った副作用（操作履歴・状態遷移・関連レコード生成など）は **永続層に残ったまま**で、次に配布されたテストに引き継がれる。
- どのエンティティが配布されるかは並列実行の順序に依存するため、汚染されたエンティティに当たるかどうかが非決定的になり flaky になる。

### 前提のすれ違い（本質的な罠）

- テストは状態の一部だけを制御して「初期状態 / クリーン」を表現したつもりでも、検証対象の分岐が **別の蓄積状態** で決まる場合、その別状態を制御できていない。
- テストが操作 A を「しない」ことで初期状態を表現したつもりでも、分岐を決める述語が依存するのは別の操作 B の履歴である、というすれ違いが起きる。テストは B を制御していないため、再配布されたエンティティに B の痕跡が残っていると前提が崩れる。
- テストのコメント / 命名に書かれた「○○しなければ初期状態になる」という前提が、実際の判定条件と食い違っていることが多い。

### 抽象化した発生シナリオ

1. プール `Get()` がエンティティ E を返す（過去のテストが使った同じ実体）。
2. ある機能は「E が初期状態のときだけ経路 X を通る」。初期状態か否かは述語 P（例: 過去に状態遷移を伴う操作がされていないか）で決まる。
3. テストは経路 X に仕込んだ検証（例: 依存先がエラーを返したらそれを呼び出し元へ伝播する）を確認したい。そのため「E を初期状態にしたい」が、実際に制御したのは P が参照しない別の属性だった。
4. 過去のテストで E に状態遷移が起きていると P=false。経路 X を通らず、仕込んだ検証が一度も実行されないまま成功扱いになる（偽陰性）。配布されるエンティティ次第で成否が変わるため flaky になる。

## 解決策

### クリーンな state を前提とするテストは専用の新規エンティティを生成する

- プールから取得（再利用）せず、テストごとに **まっさらなエンティティを生成** する。
- プールは「state を問わない使い捨て用途」（識別子や認証コンテキストが取れれば十分で、エンティティの履歴に依存しないケース）に限定する。

### 判定基準を実装ではなくドメイン契約として正確に把握する

- 「初期状態 / クリーン」が **何で決まるか**（操作履歴・ステータス・関連レコードの有無）を特定し、その全要素を制御（またはクリーンに）する。
- テストのコメントや前提が一面的な思い込みになっていないか確認し、誤っていれば是正する。

## 教訓

1. 共有フィクスチャプールは ID を再利用するため、エンティティに副作用が蓄積する。「クリーンな初期状態」を保証しない。
2. 「エラーになるはずが OK が返る / 成功するはずが失敗する」謎 flaky は、まず使用エンティティが残留状態（操作履歴・状態遷移・関連レコード）を持っていないか疑う。
3. テストが制御しているのは state の一部にすぎないことがある。検証分岐を左右する**真の条件**を特定し、それを制御する。
4. プールは「状態に依存しない使い捨て」用途に限定し、状態前提のテストは専用にエンティティを生成する。
5. 単体で成功するテストが全体並列実行で失敗する場合、テスト間の状態リーク（共有データの上書きだけでなく、共有エンティティの状態蓄積）も疑う。
