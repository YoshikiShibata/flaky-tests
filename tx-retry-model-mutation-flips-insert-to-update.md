# トランザクション再試行で使い回すモデルへの ID 書き戻しが INSERT を UPDATE に転落させる

## 概要

クライアント管理のトランザクション（競合時にクロージャを自動再実行するタイプ）の中で、
**外部から渡された可変オブジェクトを保存し、その保存処理が採番した ID をオブジェクトへ書き戻す**
実装があると、再試行時に不整合が起きる。

- 保存処理は「ID が空なら INSERT、埋まっていれば UPDATE」という典型的な分岐を持つ。
- 1 回目の試行で INSERT し、採番した ID を呼び出し元のオブジェクトへ書き戻す。
- トランザクションが競合で Abort されると、1 回目の書き込みは**破棄**される。
- しかし書き戻された ID はオブジェクトに**残る**（クロージャ外のオブジェクトはリセットされない）。
- 再試行では ID が埋まっているため保存処理が **UPDATE を選ぶ**。対象行は存在しない（1 回目の
  INSERT は破棄済み）ため、`row not found` で失敗する。

単発実行では競合が起きず 1 回で commit するため通る。**並行負荷が高まって初めて Aborted 再試行が
発火し、間欠的に失敗する** ため、フレークとして観測される。

## 前提知識：クライアント管理のトランザクションリトライ

一部の分散 DB クライアントの read-write トランザクションは、競合（Aborted 等）が起きると
**渡したクロージャ（コールバック）を丸ごと再実行**して自動リトライする。このとき:

- クロージャが**捕捉しているクロージャ外の変数・オブジェクトはリセットされない**。
- 1 回目の試行でそれらに加えた変更（append・加算・マップ追加・**オブジェクトの状態変更**）は
  再実行時にそのまま残る。

この落とし穴は次の記事で体系的に整理されている（本事例は「オブジェクト状態変更が固定化する」
パターンの実例に相当する）:

- Yoshiki Shibata「Google Cloud Spanner トランザクションリトライの落とし穴」
  https://note.com/yoshiki_shibata/n/n080f431d4ad7
  - 危険パターン: (1) スライスへの append、(2) 数値加算、(3) マップへの追加、
    (4) ループ制御変数の更新、(5) **オブジェクトの状態変更**
  - 安全策: トランザクション開始時に変数をリセット／クロージャ内でローカル宣言／
    一時変数経由で外部を更新／オブジェクトのコピーを使う

本事例は (5) に該当し、「保存処理がオブジェクトへ ID を書き戻す」という一見無害な副作用が、
再試行をまたいで固定化して INSERT/UPDATE の判定を反転させる。

## 発生条件

以下が揃うと発生する:

1. 保存処理が「ID の有無で INSERT / UPDATE を分岐」し、INSERT 時に採番した ID を
   **引数のオブジェクトへ書き戻す**。
2. その保存処理を、**トランザクションのコールバック内**で呼ぶ。
3. コールバックに渡すオブジェクトが**コールバックの外で生成され、再試行をまたいで使い回される**
   （＝クロージャが捕捉している）。
4. トランザクションが競合で Abort され、クライアントがコールバックを再実行する。

補足: 同じコールバック内でも、**毎回コールバック内で新規生成されるオブジェクト**は
（毎回 ID が空→ INSERT に一貫するため）影響を受けない。使い回されるオブジェクトだけが転落する。
この非対称性が原因の特定を難しくする。

## なぜ発見が遅れるか

- **単発・低負荷では Abort が起きない**ため 1 回で commit し、常に成功する。並行度が上がって
  初めて Aborted 再試行が発火し、間欠的に失敗する。
- 「同じテストが 1 回目の実行では通り、負荷の高い後続実行で落ちる」という形で観測されるため、
  テスト自体のフレークやデータ汚染と誤認しやすい（ID は毎回新規採番されるためデータ汚染ではない）。
- 失敗は最終的に `row not found` として現れ、直感（「今 INSERT したばかりの行が無いはずがない」）と
  反する。ログに「1 回目 INSERT → 2 回目 UPDATE（同一 ID）→ NotFound」が並んで初めて再試行と気づく。

## 具体例（一般化）

### 保存処理（ID を書き戻す）

```go
func (r repo) Save(ctx context.Context, e *Entity) error {
    var m *Mutation
    if e.ID == "" {
        m = insertMutation(e.assignNewID()) // 新規 ID を採番
    } else {
        m = updateMutation(e)               // 既存とみなして UPDATE
    }
    if err := r.apply(ctx, m); err != nil {
        return err
    }
    *e = *fromStored(e) // ★ 採番した ID を呼び出し元オブジェクトへ書き戻す
    return nil
}
```

### 呼び出し側（使い回すオブジェクトをコールバック内で保存）

```go
// entities はコールバックの外で生成され、再試行をまたいで使い回される
if err := tx.RunReadWrite(ctx, func(ctx context.Context) error {
    for _, e := range entities {
        if err := r.Save(ctx, e); err != nil { // 1回目: INSERT + ID 書き戻し
            return err
        }
    }
    // ... 他の書き込み ...
    return nil
}); err != nil {
    return err
}
```

### 観測される失敗（一般化）

```
[attempt 1] insert Entity id=X
[attempt 1] ... 別の書き込み ...
            → commit ABORTED（並行競合）。attempt 1 の mutation は破棄される
[attempt 2] update Entity id=X      # ← ID が残っているため UPDATE に転落
            → NotFound: row {id=X} not found   # attempt 1 の INSERT は破棄済み
```

## 対策

いずれも「コールバックを何度実行しても同じ結果になる（冪等）」ようにするのが目的。

1. **コールバック先頭で使い回すオブジェクトの状態を復元する**（今回採用した最小修正）。
   トランザクション前に元の ID を退避し、各試行の冒頭で書き戻す:

   ```go
   originalIDs := make([]string, len(entities))
   for i, e := range entities {
       originalIDs[i] = e.ID
   }
   if err := tx.RunReadWrite(ctx, func(ctx context.Context) error {
       for i := range entities {
           entities[i].ID = originalIDs[i] // 各試行を tx 前の状態から始める（冪等化）
       }
       // ... Save ループ ...
   }); err != nil { ... }
   ```

2. **コールバック内で使う値をローカルに閉じ込める**。使い回しオブジェクトを直接渡さず、
   毎試行コピーを作って保存する（記事の「オブジェクトのコピーを使う」）。

3. **保存処理が引数を変異させない設計にする**。ID の書き戻しをやめ、新規／既存の判定を
   ID の有無ではなく明示フラグや呼び出し前に確定した情報で行う。ただし保存処理の契約変更は
   他の呼び出し元に波及するため影響範囲を要確認。

4. **INSERT/UPDATE を Upsert（InsertOrUpdate）に統一できる場合は統一する**。再試行しても
   結果が変わらず、行の有無に依存しない。ドメイン上「必ず新規」「必ず既存」が保証できる箇所に限る。

補足: 同じコールバック内でも「毎回新規生成されるオブジェクト」は元から冪等なので手を入れない。
修正対象は「使い回されて状態が持ち越されるオブジェクト」だけに絞る。

## チェックリスト

- [ ] read-write トランザクションのコールバックは、複数回実行されても同じ mutation を生む冪等な実装か。
- [ ] コールバックが捕捉している変数・オブジェクトに、試行内で加えた変更が次の試行へ持ち越されないか
      （append / 加算 / マップ追加 / ループ変数 / **オブジェクトの状態変更**）。
- [ ] 「ID の有無で INSERT/UPDATE を分岐する保存処理」を、**採番 ID を書き戻すオブジェクト**に対して
      トランザクション内で呼んでいないか。呼ぶなら試行冒頭で状態を復元しているか。
- [ ] コールバック内で新規生成されるオブジェクトと、外から使い回されるオブジェクトを区別できているか。
- [ ] 低負荷では通り高負荷で間欠失敗する `row not found` を、テストのフレークと決めつけず
      「再試行の非冪等」を疑ったか（ログに同一 ID の insert→update が並んでいないか）。

## 教訓

- **自動リトライは「コールバックが何度でも実行される」という前提とセット**。便利さの裏で、
  クロージャが触る外部状態はすべて冪等でなければならない。
- **「保存時に採番 ID を引数へ書き戻す」ような一見無害な副作用が、再試行をまたいで固定化すると
  制御フロー（INSERT/UPDATE 判定）を反転させる**。副作用のあるヘルパーをトランザクション内で
  使い回すときは特に注意する。
- **間欠的な `row not found`／`already exists` は、テストのフレークではなく再試行非冪等の兆候**の
  ことがある。並行度を上げた反復実行はこの種のバグを炙り出す有効な手段。
- テスト側が正しく（新規 INSERT 経路を）検証していて落ちる場合、修正対象は本番実装側であることが多い。
