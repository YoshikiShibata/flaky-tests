# 逐次ループの並行化におけるループキャリー依存性の見落とし

## 概要

テストのセットアップで逐次ループを goroutine による並行ループに変換した際、特定のイテレーション間に暗黙の実行順序依存（ループキャリー依存性）が存在することを見落とし、Flaky Test が発生する現象。

逐次実行では偶然保たれていた実行順序が、並行実行では保証されなくなる。goroutine のスケジューリングにより、多くの場合は期待通りの順序で実行されるため、混入から長期間発生しないことがある。

## 発生条件

1. テストのセットアップで複数のレコードを逐次ループで作成している
2. 性能改善のためにループを goroutine で並行化する
3. 一部のイテレーション間に、**作成順序（タイムスタンプの前後関係）** に依存する検証ロジックが存在する
4. 並行実行により作成順序が不定になり、検証が失敗する

## なぜ発見が遅れるか

- goroutine はループの `i=0, 1, 2, ...` の順に起動される
- 処理量が同じイテレーション同士では、先に起動された goroutine が先に完了する傾向がある
- この傾向により、逐次実行時と同じ順序で完了することが**多い**
- マシンの負荷やスケジューラの挙動が変わったときだけ順序が逆転し、失敗する

## 具体例

### 状況

API が返すレコードのソート順を検証する E2E テスト。100件のレコードを作成し、API のレスポンスが期待通りのソート順になっているかを検証する。

ソート条件は複数の属性で構成される（属性A → 属性B → ... → タイムスタンプ）。上位の属性が同値の場合、次の属性で比較し、全属性が同値の場合は DB の作成タイムスタンプでタイブレイクされる。

テストデータの設計上、`i%10==0` と `i%10==6` のレコードは全ての属性が同値になるため、タイムスタンプによるタイブレイクが発生する。

### 逐次実行時

```go
for i := 0; i < 100; i++ {
    createRecord(i) // i=0 が先に完了 → 古いタイムスタンプ
                    // i=6 が後に完了 → 新しいタイムスタンプ
}
```

`i=0` は常に `i=6` より古いタイムスタンプを持つ。API はタイムスタンプ降順でソートするため、`i%10==6` が `i%10==0` より先に返される。検証は常に成功する。

### 並行化後

```go
for i := 0; i < 100; i++ {
    go func(i int) {
        createRecord(i)
    }(i)
}
```

goroutine の完了順が不定のため、`i=6` が `i=0` より先に完了することがある。この場合、`i%10==6` のタイムスタンプが `i%10==0` より古くなり、API のソート順が逆転して検証が失敗する。

### 解決策

依存関係のあるイテレーションだけを別の Wave に分離する:

```go
// Wave 1: i%10==6 以外を並行に作成（i%10==0 を含む）
for i := 0; i < 100; i++ {
    if i%10 == 6 {
        continue
    }
    wg.Add(1)
    go func(i int) {
        defer wg.Done()
        createRecord(i)
    }(i)
}
wg.Wait()

// Wave 2: i%10==6 を並行に作成（Wave 1 完了後なのでタイムスタンプが確実に新しい）
for i := 0; i < 100; i++ {
    if i%10 != 6 {
        continue
    }
    wg.Add(1)
    go func(i int) {
        defer wg.Done()
        createRecord(i)
    }(i)
}
wg.Wait()
```

100件中90件は Wave 1 で、10件は Wave 2 で並行実行されるため、性能への影響は最小限。

## 一般化したパターン

| 項目 | 内容 |
|------|------|
| 依存の種類 | イテレーション間の暗黙的な実行順序依存（タイムスタンプ、連番、状態遷移など） |
| 混入契機 | 逐次ループの並行化（性能改善） |
| 発見の遅れ | goroutine のスケジューリング傾向により、多くの場合は期待通りに動作する |
| 検出方法 | ループ内のデータ間に順序依存がないか確認する。特に、検証ロジックが作成順序に依存している場合は要注意 |
| 解決方法 | 依存関係のあるイテレーションを別の Wave に分離し、Wave 間の `Wait()` で順序を保証する |

## 並行化チェックリスト

逐次ループを並行化する前に、以下を確認する:

1. **各イテレーションは独立か？** — 他のイテレーションの結果（タイムスタンプ、ID、状態）に依存していないか
2. **検証ロジックは作成順序に依存していないか？** — ソート順の検証で、作成時刻の前後関係を暗黙に仮定していないか
3. **依存がある場合、どのイテレーション間か？** — 全体を逐次に戻すのではなく、依存のあるペアだけを分離する

## 関連概念

- **ループキャリー依存性（Loop-Carried Dependency）**: コンパイラ最適化の分野で、ループの異なるイテレーション間にデータ依存がある状態。並列化の障壁となる。
- **Happens-Before 関係**: 並行プログラミングにおいて、ある操作が別の操作より前に完了することが保証される関係。`wg.Wait()` は happens-before 関係を確立する。
