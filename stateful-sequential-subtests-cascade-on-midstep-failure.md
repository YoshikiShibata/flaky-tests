# 状態共有する逐次サブテストの途中失敗によるカスケード失敗

## 概要

`t.Run` でサブテストの体裁を取っているが、実体は同一エンティティに対して状態を累積していく「順序固定・状態共有の逐次シナリオ」であるテストにおいて、途中の 1 ステップがフレークで失敗すると、以降の全ステップが前提状態の崩れによって連鎖的に失敗（off-by-one 等）し、本来の原因箇所が埋もれて分かりにくくなる現象。

`t.Fatalf` は呼び出し元の goroutine しか終了させないため、サブテスト内で `t.Fatalf` を呼んでも親ループは止まらず、後続ステップが「壊れた前提状態」のまま走り続ける点が落とし穴。

## 発生条件

1. テーブル駆動風に `for _, test := range []testInput{...}` でケースを回している
2. 各ケースを `t.Run(test.name, ...)` でラップしている（＝サブテストの体裁）
3. しかし各ケースは **同一エンティティに対して状態を累積** しており、N 番目のケースは「N-1 番目までが成功している」ことを前提に検証する
4. サブテスト内では `t.Parallel()` を呼んでいない（＝定義順に逐次実行される）
5. いずれかのケースがフレーク（負荷起因の `DeadlineExceeded`、タイムアウト等）で失敗する

## なぜカスケードするか — `t.Fatalf` の終了スコープ

`t.Fatalf` は内部で `runtime.Goexit()` を呼び、**現在の goroutine だけ** を終了させる。`t.Run` はサブテストを別 goroutine で実行するため、サブテスト内の `t.Fatalf` が終了させるのは **そのサブテストの goroutine のみ**。

```go
for _, test := range cases {
    t.Run(test.name, func(t *testing.T) {
        res, err := call(ctx, ...)
        if err != nil {
            t.Fatalf(...) // ← この t はサブテストの *testing.T。
                          //    Goexit でこのサブテストは終わるが…
        }
        // assert...
    })
    // ↑ t.Run は false を返すだけ。親ループは次のケースへ進む！
}
```

その結果：

- 失敗したケース自身はそこで止まる（assert まで進まない）
- しかし **親ループは継続** し、後続ケースが「失敗ステップ分だけずれた状態」で実行される
- 状態を累積する設計のため、後続は軒並み off-by-one 等で失敗し、ログが二次失敗で埋まる
- 本来の起点（最初のフレーク失敗）が大量の連鎖失敗に埋もれて見つけにくい

## 具体例

同一エンティティに対して操作を N 回繰り返し、毎回「残り回数 = 上限 − 累積実行回数」を検証する逐次シナリオを考える。各ケースは `t.Run` で区切られているが、状態（累積実行回数）は前のケースから引き継がれる。

フルラン時、5 回目の RPC が負荷起因で `DeadlineExceeded`（サーバ側はトランザクションが context canceled でロールバック ＝ 未コミット）になったとする。

```
xxx_test.go:809: call failed: rpc error: code = DeadlineExceeded
   ↓ 5回目が未コミット ＝ 累積回数が 1 少ないまま後続へ
xxx_test.go:854: remaining count is different. want: 4, got: 5  (6th)
xxx_test.go:854: remaining count is different. want: 3, got: 4  (7th)
... 以降すべて off-by-one
xxx_test.go:865: reward amount is different. want: 0, got: 100000 (境界条件の崩れ)
```

単独再実行では PASS。＝ 根本原因は負荷起因のフレークであり、ロジックバグではない。だが連鎖失敗のせいで「どれが本物の失敗か」がログから読み取りにくかった。

### 解決策

逐次シナリオは「1 ステップ失敗したら以降は無意味」なので、ヘルパーに `t.Run` の戻り値を返させ、失敗時にループを `break` する。

```go
// ヘルパーが t.Run の合否を返すようにする
func runStep(t *testing.T, ...) bool {
    return t.Run(test.name, func(t *testing.T) {
        // ...
    })
}

for _, test := range cases {
    // 同一エンティティに状態を累積する逐次シナリオ。順序固定・状態共有のため
    // 個別実行・順序入れ替え不可。途中失敗は連鎖失敗になるので打ち切る。
    if !runStep(t, ...) {
        break
    }
}
```

これにより失敗箇所が起点 1 点に絞られ、原因（最初のフレーク）が一目で分かる。検証ロジック自体は変えない。

> 注意：これは **独立したテーブル駆動テストには適用してはいけない**。各ケースが独立なら 1 件失敗しても残りは実行すべき（カバレッジを落とさない）。`break` が正しいのは「状態共有・順序依存の逐次シナリオ」に限る。

## 一般化したパターン

| 項目 | 内容 |
|------|------|
| 症状 | 1 つのフレーク失敗が、後続サブテストの大量の二次失敗（off-by-one 等）を誘発し、原因が埋もれる |
| 根本原因 | `t.Fatalf`/`runtime.Goexit` はサブテスト goroutine しか止めず、状態共有する親ループが「壊れた前提」で継続する |
| 混入契機 | 逐次シナリオを可読性・レポート単位のために `t.Run` で区切ったが、独立テストと同じループ構造のままにした |
| 対処 | ヘルパーが `t.Run` の戻り値を返し、逐次シナリオのループは失敗時 `break`。逐次前提である旨をコメントで明示 |
| 非対象 | 独立したテーブル駆動テスト（こちらは 1 件失敗でも残りを実行すべき） |

## 設計判断チェックリスト

`for ... { t.Run(...) }` を書くとき、次を確認する：

1. **各ケースは独立か、状態を累積する逐次シナリオか？**
   - 独立 → 1 件失敗しても継続（`break` しない）。可能なら `t.Parallel()` も検討
   - 逐次・状態共有 → 失敗時 `break`。順序固定・個別実行不可である旨をコメントで明示
2. **逐次シナリオを「個別実行できる独立サブテスト」にしたいか？**
   - したいなら、ケースごとに専用エンティティを立て、必要な前提状態を各ケースで事前セットアップする（セットアップコストは増える）
3. **`t.Fatalf` の終了スコープを理解しているか？**
   - サブテスト内の `t.Fatalf` は親ループを止めない。止めたいなら戻り値で制御する

## 関連

- `t.Fatalf` → `runtime.Goexit()`：現在の goroutine のみ終了。`t.Run` は別 goroutine でサブテストを実行するため、親には伝播しない
- フレーク自体の根本対処は別問題（負荷・タイムアウト）。本パターンは「フレークが起きたときに原因を見えやすくする」ための耐性向上策
