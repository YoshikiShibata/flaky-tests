# エミュレータの直列化に隠されていた並列サブテストのデータ競合（Emulator では出ず 本番相当エンジンで顕在化）

## 概要

`t.Parallel()` を付けた 2 つのサブテストが、外側スコープの共有変数（結果格納用の
`err` など）に両方から書き込むデータ競合。コード上は作成当初から存在していたが、
テスト用 Spanner エミュレータ（cloud-spanner-emulator）上では長期間 `-race` でも
検出されず、実 Spanner エンジンである本番相当エンジン（Spanner Omni）に切り替えて
初めて `race detected during execution of test` として顕在化した。

エミュレータが read-write トランザクションを**直列化**するため、2 つのサブ
テストの該当処理が実時間上でほぼ同時に並行実行される状況が生まれず、周囲に
豊富に存在する同期操作（後述）と相まって 2 つの書き込みが常に happens-before
で順序付いていた。本番相当エンジンは本物の並行実行を行うため、2 つの書き込みが
同期を挟まずに並行するタイミングが生じ、レースが検出された。

さらに Go の race detector は競合発火時点で並行実行中の全テストを FAIL させる
ため、**原因は 1 件のデータ競合なのに多数の無関係なテストが一斉に落ちて**
見え、原因が埋もれやすい。

## 発生条件

1. 1 つのテスト関数内で `err`（や結果格納用の一時変数）を関数スコープに宣言
   （`_, err := ...`）している
2. その関数内の複数のサブテストが `t.Parallel()` で並列化されている
3. 各並列サブテストが、宣言済みの共有 `err` に `=`（`:=` ではなく）で書き込む
   （典型的には `_, err = someClient.SomeRPC(...)`）
4. `-race` 付きで実行している
5. サブテストの処理経路に、DB トランザクションや gRPC 呼び出しなど**共有資源
   を介した同期**が多数含まれる

条件 3 が競合の本体。`=` が外側の共有変数を指すことに気づかず、各サブテストの
ローカル変数のつもりで書いてしまうのが典型的な混入経路。

## なぜ発見が遅れるか（Emulator では出ず 本番相当エンジンで出た理由）

### Go race detector は動的・happens-before ベース

`-race`（ThreadSanitizer）は静的解析ではなく、**その実行で実際に発生した**
アクセスだけを見て、「同一アドレスへの 2 アクセス（片方以上が write）に
happens-before 順序が付いていない」場合にのみ報告する。バグがコードに存在
しても、その実行でたまたま順序が付けば報告されない（検出は確率的）。

### エミュレータの直列化 → 2 サブテストが同時並行にならない

cloud-spanner-emulator は RW トランザクションを実質シングルスレッドで**直列化**
する（既知の制約）。結果として、2 つのサブテストの「RPC → `err` 書き込み」
区間が実時間上でほぼ同時に走ることがない。

この種の E2E は実行中つねに**共有同期プリミティブ**を叩いている：

- テストユーザ生成のたびに呼ばれる共有カウンタ（例：ID 採番用の
  `atomic.AddInt64(&counter, 1)`）— atomic 操作は Go メモリモデル／TSan では
  **同期操作**で、この変数上の全 atomic に全順序（happens-before エッジ）が付く
- 共有 gRPC コネクション経由の RPC（応答受け取りの channel、トランスポートの
  mutex など）

実行が直列化されると、これらの同期操作が **1 本の happens-before 鎖**に綺麗に
並び、2 つの `err` 書き込みが自動的に順序付く → TSan は「順序あり」と判断し
報告しない。「直列化＝並行しない」が、豊富な同期の存在ゆえに「happens-before
で順序付く」にそのまま帰着する。

> 一般論としての注意：周囲に同期が一切無ければ、直列実行でも TSan は競合を
> 報告しうる（TSan は壁時計の重なりではなく同期エッジの有無で判定する）。
> 本件は同期が豊富なので、直列実行が確実に順序付けを生み、検出を免れていた。

### 本番相当エンジンは本物の並行実行 → 順序が切れて検出

Spanner Omni は multiplexed session で RW トランザクションを**並行実行**する。
2 つのサブテストが実際に同時進行し、2 つの `err` 書き込みの間に同期操作が
挟まらない実行が生じる → happens-before が切れて並行アクセスと判定 → 検出。

ただし依然として確率的で、本事例でも連続ローカルループのうち何度も通過し続け、
**数回目で初めて**発火した。「本番相当エンジンだから必ず出る」わけではない。

## 具体例

### 競合していたコード（一般化）

```go
func TestParallelSubtests_OK(t *testing.T) {
    ...
    // 関数スコープで err を宣言
    _, err := someClient.SomeSetupRPC(ctx, &pb.SomeSetupRequest{...})
    ...

    t.Run("case A", func(t *testing.T) {
        t.Parallel()
        ...
        // ★ 外側の共有 err に書き込み
        _, err = jobs.SomeJob(t, jobClient, ctx, argsA...)
        ...
    })

    t.Run("case B", func(t *testing.T) {
        t.Parallel()
        ...
        // ★ 同じ共有 err に書き込み（競合）
        _, err = jobs.SomeJob(t, jobClient, ctx, argsB...)
        ...
    })
}
```

### race detector の出力（要点・一般化）

```
WARNING: DATA RACE
Write at 0x00c0........ by goroutine NNNN:
  ...TestParallelSubtests_OK.funcX() ... some_test.go:<case A の書き込み行>
Previous write at 0x00c0........ by goroutine MMMM:
  ...TestParallelSubtests_OK.funcY() ... some_test.go:<case B の書き込み行>
```

2 つの並列サブテスト（別々の goroutine）が同一アドレスに書き込んでいる。

### 大量の巻き添え FAIL

race detector は競合発火時に並行実行中の全テストを FAIL させるため、原因の
サブテスト以外に多数のテストが `testing.go:....: race detected during
execution of test` で連鎖 FAIL する。**`WARNING: DATA RACE` は 1 件だけ**で、
それが唯一の原因である場合が多い。この点を最初に押さえると調査が早い。

## 対策

各並列サブテスト内で外側の共有 `err` を使わず、**ローカル変数にする**
（`=` → `:=`）だけで構造的に競合が消える。

```go
    t.Run("case A", func(t *testing.T) {
        t.Parallel()
        ...
        _, err := jobs.SomeJob(...) // := でサブテスト固有に
        ...
    })
```

検出できたか否かに依存せず競合そのものを除去できるため、これが正しい対処。
再実行で「たまたま通る」ことに頼らない。

## 潜在期間についての一般的注意

- この種の競合は、`t.Parallel()` サブテストと共有 `err` への `=` 書き込みが
  揃った時点（多くはテスト作成時）から**潜在**する。その後のリファクタで
  周辺行が移動すると `git blame` 上は後発コミットに付け替わって見えることが
  あるが、競合パターン自体はより古くから存在していることが多い。混入時期は
  「宣言と `=` 書き込みが並列サブテストに揃った最初のコミット」で判断する。

## 同じ直列化マスキングの別パターン：トランザクション内 `time.Now()` による時刻衝突

「エミュレータの RW トランザクション直列化が、並行実行前提のバグを覆い隠す」
という本記事の骨子は、**共有変数のデータ競合に限らない**。同じマスキングは
**壁時計タイムスタンプの衝突**でも起こる。

### 仕組み

エンティティの `CreateTime` 等を、リポジトリの書き込み処理内で
`time.Now()` によって設定し、**その書き込みが `RunInTransaction` の
コールバック内**で呼ばれているケースを考える（＝ `time.Now()` が
トランザクションの「中」で走る。これ自体が後述のアンチパターン）。

```go
// リポジトリ内（トランザクションのコールバックから呼ばれる）
func (e *entity) setTimestamp() {
    now := time.Now()          // ★ tx コールバック内で壁時計を読む
    if e.CreateTime.IsZero() {
        e.CreateTime = now
    }
    e.UpdateTime = now
}
```

- **エミュレータ**：RW トランザクションを直列化するため、多数の並列リクエストの
  コールバックが順番に実行される。各コールバック内の `time.Now()` は、先行
  トランザクションの処理時間（read + commit、数ms）ぶん**時間的に離れる** →
  タイムスタンプは互いに**異なる値**になり衝突しない。
- **本番相当エンジン（Omni）**：別行を触る RW トランザクションを**本当に並行実行**
  するため、コールバック内の `time.Now()` が**ほぼ同時**に走る → 壁時計の分解能が
  粗いと**同一値に衝突**する。

### なぜ問題になるか

タイムスタンプの一意性に依存する下流ロジックが、**並行実行するエンジンでだけ**
壊れる。典型は **非一意なソートキー（`CreateTime`）に tie-breaker の無い keyset
ページネーション**で、同値タイムスタンプがページ境界をまたいだ瞬間に行を静かに
取りこぼす（→ 別記事
[keyset ページネーションの tie-breaker 欠如](./keyset-pagination-missing-tiebreaker.md)）。
セットアップが一覧を全ページ走査して特定行を探す構造だと、`not found` で
間欠失敗する。

### 重要な訂正

「時刻衝突は壁時計の分解能に由来する**クライアント側の性質で DB エンジンに
非依存**」と説明されがちだが、**`time.Now()` がトランザクションの中にある場合は
これは誤り**。エンジンのトランザクション直列化が `time.Now()` の実行タイミングを
gate するため、**衝突の発生自体が「並行実行か直列化か」に支配される**。

### 対処

- 時刻は **トランザクション開始前に一度だけ `time.Now()` で取得し、クロージャに
  渡す**（`RunInTransaction` の外で確定させる）。これは
  「コールバックはリトライで再実行されるため tx 内で `time.Now()` を呼ばない」
  という一般規約とも一致し、**(1) リトライでのタイムスタンプ変動、(2) 衝突の
  エンジン依存性**の両方を同時に解消する。
- そのうえで、非一意なソートキーには必ず一意な tie-breaker を添える
  （タイムスタンプは並行書き込みで**必ず衝突し得る**前提で設計する）。

### 教訓（このパターン）

**「エミュレータで出ないが本番相当エンジンで出る」現象を見たら、まず
「エミュレータの直列化が何を覆い隠していたか」を疑う。** 覆い隠される対象は
共有変数のデータ競合とは限らず、トランザクション内 `time.Now()` の時刻衝突の
ように、**並行実行して初めて顕在化する非決定性**全般に及ぶ。

## 教訓・一般化

- `t.Parallel()` サブテストからは、**外側スコープの可変変数に絶対に書き込まない**
  （`=` を見たら疑う。結果格納は必ず `:=` でサブテスト内ローカルに）。
- 低頻度データレースは `-race` でも取りこぼす。**検出できた＝バグが今出た**では
  なく、多くは以前から潜在していたものが環境変化（DB エンジンの直列化 →
  並行化、負荷、スケジューリング）で顕在化しただけ。
- エミュレータの緩い挙動（RW 直列化・行順序・分離レベル）は潜在的な非決定性を
  覆い隠す。本番相当エンジンでの定期実行は、こうした潜在バグを炙り出す価値が
  ある。関連: [エミュレータ vs 本番DBエンジンの挙動差](./emulator-vs-production-db-divergence.md)、
  [長時間ランニングテストの重要性](./importance-of-long-running-tests.md)。
- 大量の巻き添え FAIL を見たら、まず `WARNING: DATA RACE` の件数を数える。
  1 件なら原因も 1 件。
