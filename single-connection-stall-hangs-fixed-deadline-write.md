# エミュレータへの書き込みが単一コネクションの転送スタックで稀にタイムアウトする

## 概要

テストのセットアップ（seed）で、ホスト側のテストプロセスから**コンテナで動くエミュレータ**
（GCS エミュレータ等）へ HTTP でオブジェクトを書き込む処理が、稀に固定タイムアウト
（例: 50 秒）まで待って `context deadline exceeded` で失敗する。エミュレータ自体は正常に
高速動作しており（インメモリ backend で毎秒数百リクエストを処理し続けている）、詰まっている
のは**特定の 1 本の TCP コネクションのデータ転送だけ**。原因はエミュレータの処理遅延ではなく、
**ホストとコンテナの間のネットワーク中継**（macOS の Docker Desktop 等が使うユーザーランドの
ポートフォワード proxy）が、高いコネクション生成レートのピークで**個別のフローを宙吊りにする**
ことにある。

対策は「タイムアウトを延ばす」ことではない。**短いタイムアウトで見切って、試行ごとに ctx を
張り直して新しいコネクションで再送するリトライ**。詰まりが 1 本のコネクションに孤立している
ため、新しいコネクションはほぼ即座に成功する。

## 症状

- 単独実行・短時間では再現しない。**長時間の連続ループ実行**（数回〜十数回に 1 回）でだけ落ちる
- 失敗は書き込みの `Close`（アップロード完了）で `context deadline exceeded`
- 失敗した 1 リクエストだけが固定タイムアウトいっぱい（例: 50 秒）張り付く。周囲の他リクエストは
  正常
- インメモリ backend に切り替え済み・毎回コンテナを再起動していても再発する（＝ディスク I/O や
  状態蓄積が原因ではない、という手掛かり）
- CI（コンテナ間で直接通信し中継を経由しない構成）では出ず、**ローカルのホスト→コンテナ経由**
  でだけ出る

## 最初の仮説（fsync）と、それが未検証だったこと

同じ「書き込みの `Close` が deadline 超過」という症状に対し、**先に別の仮説**で手を打っていた。

- **第 1 段階（fsync 仮説・未検証）**: filesystem backend 時代に同種のフレークが**より高頻度**で出て
  いた。エミュレータの filesystem backend は、オブジェクトごとにディスクへの `fsync` を**グローバル
  ロック保持中**に行う実装であることから、「並行書き込み＋長時間ループで fsync がボトルネック化して
  いる」という**仮説**を立て、backend を **filesystem → インメモリ**に切り替えた（fsync とディスクへの
  無制限蓄積も排除）。切り替え後、高頻度のフレークは**見かけ上収まった**。
  - ただし **fsync が真因だったことは証明されていない**。プロファイル等でロック保持時間や fsync の
    ブロックを直接測ったわけではなく、バージョン bump 直後に表面化したという**状況証拠**から立てた
    仮説にすぎない。memory 化が fsync を除いたから効いたのか、単に頻度が下がっただけなのか、あるいは
    **後述の転送スタックと同じ現象が filesystem backend の遅さで増幅されていた**のか——**切り分けて
    いない**。
- **第 2 段階（残った低頻度の再発・こちらは証明済み）**: **インメモリ backend にして・毎回コンテナを
  再起動していても**、低い頻度で同じ症状が再発した。ここで **「インメモリなのに固定タイムアウト
  いっぱい張り付く」という矛盾**（インメモリ書き込みはマイクロ秒オーダーなので数十秒の滞留は処理
  遅延ではありえない）が、backend の遅さではなく**転送のハング**を疑う手掛かりになり、**サーバ側
  アクセスログという直接証拠**で単一コネクションの転送スタックを**特定できた**（次節）。

**この対比が学びの核**: 第 1 段階の fsync は「妥当だが未検証の仮説」、第 2 段階の転送スタックは
「ログで裏を取った証明済みの真因」。**妥当そうな修正で高頻度の失敗が『見かけ上』消えても、それは
真因を証明したことにはならない**。残った低頻度の再発こそが、第 1 段階の説明が不完全（あるいは誤り）
だったことを示すサインだった。同じ症状でも頻度・条件が変わったら、前の仮説を「確定した真因」と
思い込まず、証拠を取り直す。

## 根本原因（第 2 段階の真因）

### 「エミュレータが遅い」ではなく「1 本のコネクションが止まっている」

インメモリ backend の書き込みはマイクロ秒オーダー。それが**固定タイムアウトちょうど**で落ちるのは、
処理が遅いのではなく**リクエストの応答が返っていない（＝ハング）**ことを意味する。原因はエミュレータ
の外、リクエストがエミュレータに届く／返る**転送経路**にある。

ホスト側のテストプロセスから `localhost:<port>` でコンテナのエミュレータに接続する場合、通信は
**コンテナランタイムのユーザーランドのポートフォワード中継**を必ず通る（macOS の Docker Desktop
の VM ネットワーク proxy 等）。この中継は、短命なコネクションが大量に生成されるピーク時に、稀に
**特定の 1 フローだけ**データの送受信を止めてしまうことがある。

### 決定的な切り分け: サーバ側アクセスログ

真因の確定には**エミュレータ側のアクセスログ**が要る。観測された事例では:

- 失敗したリクエストは**正常な時刻にエミュレータへ届いていた**（アクセスログの受信時刻＝ハンドラ開始が
  他の兄弟リクエストと同時刻）。しかし**ハンドラ内に数十秒滞留**してから完了していた（ヘッダは即着、
  ボディの転送が途中で止まっていた）。クライアントは先に固定タイムアウトで諦めていた
- その滞留の数十秒間、**エミュレータは毎秒数十〜数百件を処理し続けていた**。詰まっていた時間窓に
  完了したリクエストの **99% 以上が即時（< 1 秒）** で、長時間かかったのは問題の 1 件だけ

この 2 点が「エミュレータ全体のフリーズ／グローバルロック／backend 遅延／中継の全断」を**すべて否定**し、
「**特定の 1 本のコネクションだけが孤立して止まった**」ことを裏付ける。

```
# サーバ側アクセスログの読み方（概念）
[受信 T+0s] POST .../objectA  -> 完了 T+0s   200   （即時）
[受信 T+0s] POST .../objectB  -> 完了 T+0s   200   （即時）
[受信 T+0s] POST .../objectC  -> 完了 T+57s  200   （★1本だけ数十秒滞留）
# 同じ窓で他の数百リクエストは即完了 = エミュレータは生きている
```

## なぜリトライで直り、単発タイムアウト延長では直らないのか

分かれ目は**詰まりが「コネクション単位で孤立」しているか**。

- **孤立している**（本事例）: 詰まっているのは今掴んでいる 1 本だけで、**同時刻に新しく張られる別の
  コネクションは正常に通っている**（上記ログが裏付け）。
  - 単発の長いタイムアウト = 詰まった**同じ 1 本**をずっと待ち続けるだけ。一度も別コネクションを
    試さないので、待ち時間を延ばしても直らない
  - 短いタイムアウト × リトライ = 早めに見切って**新しいコネクション**で送り直す → ほぼ即成功
- **全体障害**（proxy 全断・エミュレータのデッドロック等）なら、その間どの新規コネクションも詰まる
  ので、リトライは無意味。→ **サーバ側ログで「他リクエストが捌けているか」を必ず確認**してから
  リトライを処方する

リトライの効き目は「待ち時間を延ばす」ことではなく、**「止まった 1 本を早く捨て、健全な別の 1 本に
乗り換える」**こと。統計的にも、1 本が詰まる確率は極めて低いため、連続で 2 本詰まる確率は無視できる。

## 解決策

### 短い per-attempt タイムアウト × 新コネクションでのリトライ

単一オブジェクト書き込みを共通ヘルパーに集約し、**試行ごとに ctx を張り直す**。ctx をキャンセルすると
その試行のコネクションは閉じられる（アイドルプールに戻らない）ため、リトライは必ず**新しいコネクション**
で送られる。書き込みが冪等（同一パスに同一内容の上書き）なら、リトライしても副作用はない。

```go
// 良い例: 試行ごとに ctx（＝コネクション）を張り直す短タイムアウト × リトライ
func writeWithRetry(write func(ctx context.Context) error) error {
    const perAttemptTimeout = 15 * time.Second
    const maxAttempts = 4
    var lastErr error
    for attempt := 1; attempt <= maxAttempts; attempt++ {
        ctx, cancel := context.WithTimeout(context.Background(), perAttemptTimeout)
        lastErr = write(ctx) // NewWriter(ctx) → Copy → Close を内包
        cancel()
        if lastErr == nil {
            return nil
        }
        // 経過観察のため、どの試行が失敗して張り直したかをログに出す
    }
    return fmt.Errorf("write failed after %d attempts: %w", maxAttempts, lastErr)
}
```

- per-attempt タイムアウトは、正常な書き込み（< 数秒）に対して十分な余裕を持たせつつ、止まった
  コネクションを早く捨てられる長さにする（例: 15 秒）
- **回復・リトライ発生をログに出す**と、経過観察で「実際に起きて吸収できているか」が可視化できる。
  観測された事例では、連続ループ中に発生したスタックが**すべて 2 回目（新コネクション）で即回復**した

### 補足: 発生頻度を下げたいなら keep-alive を増やす手もあるが慎重に

新規コネクションの乱立を抑える（HTTP クライアントのアイドルコネクション上限を上げて再利用を増やす）
のは、詰まりが「コネクション確立時」に相関するなら露出を減らせる。ただし本事例の詰まりは**確立済み
コネクションの転送途中**でも起きており、効果は不確実。さらに、**詰まりかけた（が閉じていない）
コネクションがプールに残って次のリクエストに配られる**裏目もある。リトライは root cause 非依存で
確実に回復するので、**リトライを主対策とし、keep-alive 調整は補助**に留めるのが安全。

## なぜ別プロトコルのクライアント（gRPC 等）では起きにくいか

同じホスト→コンテナの中継を通っていても、**プロトコルと接続の使い方**で露出が桁違いに変わる。

| | 多重化 gRPC（HTTP/2）系クライアント | HTTP/1.1 の REST/JSON 系クライアント |
|---|---|---|
| コネクション | 少数・長寿命をプールし多重ストリームで使い回す | 並行書き込みごとに新規 TCP を張り、使い捨てる |
| 新規接続の生成レート | 起動時のみでほぼ増えない | ピークで毎秒数百 |
| クライアント自動リトライ | RPC／トランザクションに内蔵＋keepalive で死活検知 | 標準では無し（自前で書く必要） |

中継が 1 フローを止める事象は**新規接続の生成量に比例**するため、接続を大量に張り捨てる HTTP/1.1 系が
突出して踏みやすく、かつ自動リトライが無ければ即失敗する。多重化 gRPC 系は「少数の長寿命接続＋
ライブラリのリトライ」で二重に守られているため、同じ中継でも表面化しにくい（**免疫ではなく露出が
小さいだけ**）。今回の対策は、この「クライアント側リトライ」を HTTP/1.1 系の書き込みにも与えるもの。

## 一般化・レビューチェックリスト

- [ ] エミュレータへの書き込みが固定タイムアウトで落ちたとき、**「エミュレータが遅い」と決めつけて
      タイムアウトを延ばす**前に、サーバ側アクセスログで「**リクエストは届いていたか**」「その間
      **他リクエストは捌けていたか**」を確認したか
- [ ] インメモリ backend なのに固定タイムアウトいっぱい張り付くなら、それは処理遅延ではなく
      **転送のハング**。原因はエミュレータの外（ホスト↔コンテナの中継）を疑う
- [ ] 症状が同じ（`Close` の deadline 超過）でも、前に打った手（例: fsync **仮説**に基づく memory
      backend 化）で高頻度が**見かけ上**消えたことを「真因を特定した」と混同していないか。前の説明が
      **未検証**なら、残った低頻度の再発は別の（あるいは同じだが未証明の）root cause かもしれない
- [ ] 詰まりが**単一コネクションに孤立**しているか（他は正常）を確認したうえでリトライを処方したか。
      全体障害ならリトライは無意味
- [ ] リトライは**試行ごとに ctx（＝コネクション）を張り直して**いるか。同じ ctx／同じ接続を使い回すと
      詰まった 1 本に固執して回復しない
- [ ] 書き込みが**冪等**でリトライ安全か
- [ ] ローカル（ホスト→コンテナ中継経由）でだけ再現し CI（コンテナ間直結）では出ない、という
      非対称を見落としていないか

## 教訓

1. **同じ症状に複数の独立した真因がありうる。**最初の妥当な修正（fsync 対策の memory backend 化）で
   高頻度の失敗が消えても、残った低頻度の再発は別物（転送スタック）だった。「直った」と思い込まず、
   頻度・条件の変化を手掛かりに次の root cause を疑う
2. **「エミュレータが遅い」と「1 本のコネクションが止まった」は別物。**インメモリ backend で固定
   タイムアウトいっぱい張り付いたら、処理遅延ではなく転送のハングを疑い、サーバ側ログで裏を取る
3. 単発タイムアウトの延長は、**孤立した 1 本のコネクションに固執し続ける**ため直らない。正しい処方は
   **短いタイムアウトで見切り、新しいコネクションで張り直すリトライ**
4. リトライを処方する前に「その間、他のリクエストは通っていたか」をログで確認する。**通っていれば孤立＝
   リトライが効く／全部詰まっていれば全体障害＝リトライは無意味**
5. ホスト→コンテナの `localhost` 接続は、コンテナランタイムのユーザーランド中継を通る。**高い接続生成
   レートのピークで個別フローが宙吊りになる**ことがある。CI（コンテナ間直結）で出ずローカルで出る
   非対称はこのサイン
6. 同じ中継でも、**多重化 gRPC（少数・長寿命・自動リトライ）は露出が小さく、HTTP/1.1 の接続使い捨ては
   露出が大きい**。プロトコルの違いが flaky の出やすさを左右する
7. 低頻度は環境要因（壁時計等）を意味しない。**狭い時間窓 × 稀な中継不調**で確率が下がっているだけで、
   設計上は常に起こりうる
