# プロセス共有キャッシュによる test-scoped fake override の無効化

## 概要

サービス側に **プロセス共有の TTL キャッシュ** が導入されている API について、E2E テストが **テストスコープの fake server override**（tid スコープなど）を設定しても、他テストが先にキャッシュを populate していると override が反映されず、テストが flaky に失敗する。

> NOTE: fake サーバーや tid（テスト識別子）の概念・設計については、書籍『[Web API設計実践入門](https://www.amazon.co.jp/exec/obidos/ASIN/4297142937/yoshikisbooks-22/)』（柴田 芳樹 著、技術評論社）を参照のこと。

## 症状

- 特定のテストを単独で実行すると常に成功する
- 全 E2E テストを並列実行すると、当該テストがランダムまたは特定順序で失敗する
- 失敗時、テストが設定した fake 応答とは異なる値（**他テストの実行時点でデフォルト fake が返した値**）に基づいた検証エラーになる
- ログを追っても、当該テストの tid を持つ fake への RPC 呼び出しが**そもそも記録されない**（cache miss しないので fake 自体が呼ばれない）
- キャッシュ TTL（例: 5分）より短い時間で複数テストが流れる CI/ローカル一括実行でのみ顕在化する

## 根本原因

この問題は以下の3つの構造的要因の組み合わせで発生する。

### 構造的要因

1. **サービス側のキャッシュがプロセス共有のシングルトン**: in-memory キャッシュが gRPC リクエスト・テスト識別子・ユーザーを問わず共通のキーで保持される
2. **テスト用 fake が tid (テスト識別子) スコープで応答を切り替える**: tid が一致しない場合はデフォルト応答にフォールバックする
3. **キャッシュは fake への RPC 直前で先回りしてヒットする**: cache miss の場合のみ fake が呼ばれる

### 発生メカニズム

1. テストプロセス起動時、fake server に **デフォルト応答** が設定される
2. **他テスト T1**（tid 上書きを設定しないもの）が走り、対象 API を呼び出す → サービス内でキャッシュミス → fake にデフォルト応答で問い合わせ → デフォルト値がキャッシュに格納される
3. **対象テスト T2** が走り、自身の tid スコープで fake override を設定 → 対象 API を呼び出す → サービス内で **キャッシュヒット**（T1 が入れたデフォルト値が返る）→ fake への RPC が発生せず、T2 の override が一度も使われない
4. T2 はデフォルト値に基づく結果を受け取り、override 前提の検証が失敗する

### tid スコープ override が成立する前提条件

`SetXxxResponseCreator(tid, ...)` 系の tid スコープ override は、**サービス側がリクエストごとに毎回 fake を呼ぶ**ことを暗黙の前提としている。サービス側が結果をキャッシュした瞬間、この前提は崩れる。

### サービス側の典型コード

```go
// サービス内
func someUseCase(ctx) ([]int64, error) {
    cached, err := cache.Get(baseTime)
    if errors.Is(err, ErrCacheNotFound) {
        // ↓ ここが呼ばれた tid に対応する fake creator が動く
        rules, err := remoteClient.ListXxx(ctx)
        cache.Put(rules, ttl)
        return rules, nil
    }
    // ↓ キャッシュヒット時はここが返るので tid に対応する fake creator は呼ばれない
    return cached, nil
}
```

## 解決策

### 案A: tid スコープ override をやめて default fake を期待値に合わせる（推奨）

テストの期待値を default fake server の応答に含めるよう設定する。プロセス共有キャッシュがどのテストの tid で populate されても、結果が常に期待を満たすようになる。

#### 修正前

```go
// テスト
func TestXxx_OK(t *testing.T) {
    tid, tctx := tid.New(ctx)
    fakeXxx.SetListXxxResponseCreator(tid, func(...) {
        return testSpecificResponse, nil
    })
    // ...
}
```

#### 修正後

```go
// fake/xxx_default.go に共有定数を切り出す
const FakeXxxIDForFooTest int64 = 1_000_000_001

func setupDefaultFake(srv *FakeServer) {
    srv.SetDefaultListXxxResponseCreator(func(...) {
        return &Response{
            Items: []*Item{
                ...
                {ID: FakeXxxIDForFooTest, ...},  // 当該テストが期待する値
            },
        }, nil
    })
}

// テスト
func TestXxx_OK(t *testing.T) {
    // tid スコープ override は不要
    // default fake が FakeXxxIDForFooTest を返す前提で組む
    db.SetUp(..., FakeXxxIDForFooTest)
    // ...
}
```

#### 適用条件

- テストが期待する値がドメイン上一意で、default に追加しても他テストへの副作用がないこと
- 追加した値が dev/prd の本物の値と区別できる、テスト専用の架空 ID であること（例: 大きな数値や予約レンジ）

### 案B: サービス側にキャッシュバイパス手段を用意する

E2E モード時にキャッシュ TTL を 0 にする、もしくはキャッシュクリア用の admin RPC を追加する。設計影響が大きいので、案 A で対処できない場合の最終手段。

### 案C: キャッシュ前提をテスト全体で揃える

すべての関連テストが同じ default を返す fake で動作することを前提に、テスト側のキャッシュ依存をなくす（fake override をそもそも使わない）。

## 予防（PR レビュー時の観点）

このパターンの本質は、技術的なバグというより **「キャッシュ導入が E2E テストの前提を破壊する」という影響範囲を、PR 著者・レビュアー双方が想像できなかった** ことにある。実際、本問題で導入されたキャッシュ PR 時点でテスト全件は pass しており、構造的破綻が顕在化したのは数日〜数週間後のテストデータ変更がトリガーだった。

「pass している」ことは「壊れていない」ことの証明にはならない。キャッシュやメモ化、シングルトンなど **「同一プロセス内で前回の結果を覚える」種類の最適化** を入れる PR では、以下の観点を設計レビューで明示的に議論すべき:

| 観点 | 確認内容 |
|---|---|
| キャッシュキーの粒度 | tid / リクエスト ID / ユーザー ID を含むか、プロセス全体で 1 キーか |
| キャッシュ有効範囲 | リクエストスコープ／ユーザースコープ／プロセス全体スコープのいずれか |
| E2E テストの fake 利用箇所 | キャッシュ対象 API に対し、`SetXxxResponseCreator(tid, ...)` など test-scoped override を使っているテストが存在するか |
| 既存 default fake の応答 | 既存テストがその default 応答に暗黙に依存していないか |
| キャッシュバイパス手段 | E2E モードで TTL=0 にする、テスト用クリア API を持つなど、テストが回せる経路があるか |

PR 説明文に「**この変更により、X というテスト群の前提が変わる**」「fake override は cache miss 時のみ反映されるため、test-scoped 設定は信頼できなくなる」といった**副作用の明示的な記述**を求めるのが現実的なゲートになる。

## 検出のヒント

このパターンが疑われる場合に確認する観点:

| 観点 | 確認内容 |
|---|---|
| 該当テスト単独実行 | 成功する → 並列実行依存の flaky |
| 失敗時のサービスログ | 当該テストの tid を持つ fake RPC のログが**ない** |
| 対象 API のサービス実装 | `cache.Get` → cache miss 時のみ remote 呼び出し、というパターン |
| キャッシュキー | プロセス全体で 1 つの key（テスト tid やリクエスト ID を含まない） |
| fake 設定箇所 | tid スコープの `SetXxxResponseCreator(tid, ...)` |
| 並列実行される他テスト | 同じ API を間接的に呼ぶテストが多数（特に default fake 応答にフォールバックするもの） |

## 教訓

1. **プロセス共有キャッシュはテスト用 fake の tid スコープを貫通する**: サービスが結果をキャッシュした瞬間、tid スコープ override は「キャッシュミス時にだけ効く」確率的な仕組みになる
2. **キャッシュ導入時に E2E テストへの影響を想像する責任は PR 著者・レビュアー双方にある**: 「pass しているから問題ない」は通用しない。キャッシュ・メモ化・シングルトンなど "前回の結果を覚える" 種類の変更は、テストの前提を後から壊す典型パターン。PR 時点で flaky 化しなくても、数日〜数週間後にテストデータ・実行順・並列数が変わった瞬間に表面化する
3. **キャッシュ導入は E2E テストの fake 設定戦略を破壊しうる**: 既存テストが pass していても、後からキャッシュを足したらテストの前提が崩れる。キャッシュ導入の PR では「fake override を使っているテスト」を全件レビューする必要がある
4. **fake のデフォルト応答はテスト間の暗黙の契約**: 複数テストが共有する fake のデフォルト値は、そのテストプロセス全体の事実上の DB スキーマと同じく重要な共有資源。変更時は全関連テストへの影響を確認する
5. **ランダム値と共有キャッシュは相性が悪い**: テストごとに `rand.Int63()` で生成した値を fake で返すと、キャッシュに入った瞬間その値はテスト境界を越えて他テストへ漏れる。テスト固有値は default fake への登録か、固定の予約レンジから取るのがよい
6. **「override が効くべき」という思い込みを捨てる**: 「tid スコープで設定したから自分のテストにしか影響しない」は、サービス実装次第で成立しないことがある。設定が反映されているか、サービス側のログ・キャッシュ実装で実際に確認する
