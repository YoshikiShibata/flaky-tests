# グローバル集計値の before/after 比較が並行書き込みで揺れる

## 概要

「ある操作が状態を変えないこと（あるいは想定どおりに変えること）」を、**全エンティティ横断の集計値
（件数・一覧・合計など）**を操作の前後でスナップショットして比較する形で検証しているテスト。集計対象が
テスト対象のエンティティに絞られておらず「グローバルに共有された観測量」になっているため、`t.Parallel()`
下で無関係な別テストが集計に寄与する行を書き込むと、before と after の間で値が揺れて、実行のたびに
成功したり失敗したりする（flaky になる）。

対象の一覧／件数 API 自体は「全件横断で返す」ことが正しい仕様（ope / 管理用途の一覧など）であるケースが
多く、**書き込み側を全部ロックするのは非現実的で誤った処方箋**。直すべきはテスト側の
**アサーションのスコープ**。

## 症状

- 単独実行では常に成功する
- 全 E2E を並列実行すると、まれに `got N+1, want N`（あるいは `N-1`）のような **±1〜数件のズレ**で落ちる
- ズレるのは常に「テスト対象と無関係なデータが 1 件混ざった／減った」方向
- 連続ループ実行の**数回〜十数回に 1 回**しか再現しない（後述のとおり窓が狭い）

## 根本原因

### 問題のある構成

- **検証テスト B**: 対象エンティティ `e`（テスト内で新規作成した自分専用のエンティティ）を用意し、
  「操作 `Op` が状態を変えないこと」を検証したい。検証手段として、**全エンティティ横断で対象行を返す
  一覧／件数 API `L`** を `Op` の前後で呼び、返ってきた**全体の集計値**（件数など）が一致することを
  assert する。`Op` 自体は集計に寄与する行を書かない。
- **無関係な並行テスト A**: 別のエンティティに対して、`L` の集計条件に一致する行を通常業務として書き込む。
  A は B のデータには一切触れない。

```go
// 悪い例: グローバル集計の before/after 比較
before := countViaGlobalListAPI(t, ctx, L)   // 全エンティティ横断の合計 = N
op(t, ctx, e)                                // 集計に寄与する行は書かない操作
after := countViaGlobalListAPI(t, ctx, L)    // 稀に N+1
if after != before {
    t.Errorf("got %d, want %d", after, before)
}
```

### API が「全件横断」であることは気づきにくい

`L` が呼び出し元のエンティティ（顧客・ユーザー・テナント等）でスコープされているように見えても、
実装を辿ると**どこにもスコープの絞り込みが無い**ことがある（handler は所有者 ID を渡さない → usecase は
種別だけで絞る → repository の `WHERE` に所有者条件が無い、など）。

**baseline の集計値が「テスト対象の規模」に対して不自然に大きいのは、スコープが広すぎるサイン。**
新規作成した 1 エンティティしか関与しないはずのテストで baseline が数百件あるなら、それは全件横断を
疑う根拠になる。

### 発生メカニズム

1. B が before スナップショットを取る（グローバル集計 = N）
2. B が検証対象の操作 `Op` を実行。これ自体は集計に寄与する行を書かない
3. **その最中に**、無関係な並行テスト A が集計条件に一致する行を commit
4. B が after スナップショットを取る（N+1）→ 不一致で FAIL

B は自分のデータを一切壊していないし、A も B のデータを壊していない。**誰も「他人のデータ」に触って
いないのに、B のアサーションが「全員の合計」を見ているせいで壊れる**——ここが
[全データ更新ジョブによるテスト間干渉](./global-data-modification-interference.md)（他人のデータを
**上書き**するので**ロック**で直す）との決定的な違い。本件では誰も上書きしていない。

## なぜ低頻度なのか

汚染が成立するのは、**before スナップショットと after スナップショットの「間」**に、集計条件に一致する
書き込みが commit された場合**だけ**。

- commit がこの窓より**前**なら → before・after 両方に反映され、差が出ない
- commit がこの窓より**後**なら → 両方に反映されず、差が出ない
- **ちょうど間**に commit されたときのみ ±1

さらに、その窓の広さは「操作 `Op` ＋2 回のスナップショット取得」のサブ秒オーダーで、かつ集計条件に
一致する行を書く並行テストは限られる。**狭い窓 × 限られた汚染源**の積で確率が下がるため、設計上は
常に壊れうるのに再現は十数回に 1 回になる。「低頻度だから環境要因（壁時計等）だろう」と早合点しない
こと——**低頻度は分離不足の否定材料にならない**。

## 解決策

### 原則: アサーションを「テスト対象のエンティティ」にスコープする

グローバル集計ではなく、**自テストが作成／所有するエンティティに属する分だけ**を観測して比較する。
一覧 API のレスポンスに所有者やエンティティの識別子が含まれるなら、クライアント側でフィルタすればよい。
これで並行テストのグローバルな書き込みから完全に独立する。

### さらに堅牢化: 件数比較ではなく「集合の同一性」比較

「変化しないこと」の検証では、件数一致は弱い（`+1 と -1` が相殺すると見逃す）。**識別子の集合**を
before/after で取り、集合が完全一致することを assert すると、追加・削除・入れ替わりのいずれも検出できる。

```go
// 良い例: 自テストのエンティティに絞り、ID 集合の同一性を比較
func listOwnIDs(t, ctx, L, ownerID) []string {
    var ids []string
    for /* 全ページ走査 */ {
        for _, row := range res.Rows {
            if row.OwnerID == ownerID {      // 自テスト分だけ
                ids = append(ids, row.ID)
            }
        }
    }
    sort.Strings(ids)
    return ids
}

before := listOwnIDs(t, ctx, L, ownerID)
op(t, ctx, e)
after := listOwnIDs(t, ctx, L, ownerID)
if diff := cmp.Diff(before, after); diff != "" {
    t.Errorf("state changed (-before +after):\n%s", diff)
}
```

（一覧が keyset（カーソル）ページングで、ソートキーが `(CreateTime, ID)` のように一意に定まるなら、
他エンティティの並行 insert でページ境界がずれても自テスト分の既存行は各 1 回ずつ確実に現れ、
二重計上・脱落は起きない。ソートキーが非一意だと別問題を招く →
[keyset ページネーションの tie-breaker 欠如](./keyset-pagination-missing-tiebreaker.md)。）

### なぜロックで直さないのか

対象 API が「全件横断で返す」のは正しい仕様。集計に寄与しうる書き込みは**多数の無関係なテストに散在**
しており、それら全部に共有ロックを課すのは並列性を大きく損ない、かつ将来のテスト追加で簡単に破れる。
グローバル集計を安定させる責務をテスト分離側に押し付けるのは筋が悪い。**観測をローカル化するほうが、
安く・確実で・将来に強い。**

## 一般化・レビューチェックリスト

- [ ] before/after スナップショット比較のアサーションで、**スナップショットの対象は「テスト対象の
      エンティティ」に絞られているか？** 全件横断の集計を見ていないか
- [ ] その一覧／件数 API は**所有者（顧客・ユーザー・テナント等）でスコープされているか、それとも
      全件横断か**を呼び出し経路（handler → usecase → repository の `WHERE`）まで辿って確認したか
- [ ] baseline の集計値が「テスト対象の規模」に対して**不自然に大きく**ないか（大きい＝スコープが広い疑い）
- [ ] 「変化しないこと」の検証で**件数一致に頼っていないか**（`+1/-1` 相殺の見逃し）。集合・識別子の
      同一性で見ているか
- [ ] `±1〜数件`・低頻度のフレークを見たら、**環境要因の前に「グローバル観測量への並行書き込み」**を疑う

## 教訓

1. **「操作が状態を変えないこと」を全体集計の不変性で検証してはいけない。**テスト対象に属する分だけを
   観測する。グローバル集計は、無関係な並行テストが自由に動かせる共有可変状態である
2. 一覧／件数 API を検証に使う前に、**それが自分のスコープを返すのか全件を返すのか**を実装で確認する。
   レスポンスに識別子があるならクライアント側で自テスト分に絞れる
3. 「誰も他人のデータを壊していない」場合でも、**アサーションのスコープが広すぎればフレークする**。
   分離の問題は「書き込み側」だけでなく「読み取り側の観測範囲」にも宿る
4. 低頻度は分離不足を否定しない。狭い時間窓 × 限られた汚染源で確率が下がっているだけで、設計上は
   常に壊れうる
5. ロックは万能ではない。API が正当にグローバルなら、**観測をローカル化する**ほうが正しい処方箋
