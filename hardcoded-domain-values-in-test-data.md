# テストデータのハードコード値とドメイン定数の不一致

## 概要

テストでDBにデータを直接挿入する際に、ドメイン定数と一致しないハードコード文字列を使用していたことで、並列実行時に他のテストが全件取得APIを呼んだ際に変換エラーが発生するFlaky test。

## 症状

- 特定のテストを単独で実行すると常に成功する
- 全E2Eテストを並列実行すると、特定のテストがランダムに失敗する
- APIの内部エラー（例: `unexpected xxx: INVALID_VALUE`）が発生する
- CIではテストが関数名プレフィックスで別ジョブに分割されているため再現せず、ローカルの一括実行でのみ発生する

## 根本原因

この問題は以下の2つの構造的要因の組み合わせで発生する。

### 構造的要因

1. **機能単位でテストが分割作成される**: webhook機能のテスト担当者は、自分のテストに必要な前提データを「自分のテストが通ればよい」レベルで作成する。全件取得APIのテストは別の人・別のタイミングで作成されるため、データの正当性が横断的に検証されない
2. **Spannerを全テストで共有している**: 全テストが同一のSpannerインスタンスに対して実行されるため、あるテストが挿入したデータが他のテストから見える

### 問題のあるテスト構成

1. **テストA**: DBに直接データを挿入する際、ドメイン定数とは異なるハードコード文字列を使用（例: `"A8NET"` vs ドメイン定数 `"a8net"`、`"PUBLIC"` vs `"published"`）
2. **テストB**: 全件取得APIを呼び出し、取得したデータをドメイン定数ベースで変換・検証する

### 発生メカニズム

1. テストAが不正なハードコード値でDBにデータを INSERT
2. テストBが並列で実行され、全件取得APIを呼び出す
3. APIがSpannerから全レコードを読み出し、readmodelに変換する
4. テストAが挿入した不正値のレコードが、ドメイン定数とのswitch文マッチに失敗
5. APIが Internal Error を返し、テストBが失敗

### テストAが単独で成功する理由

テストA自身はwebhook等のエンドポイントをテストしており、挿入したデータの変換処理を通らないため、ハードコード値の不一致に気づかない。

## 解決策

### DBへの直接挿入時にドメイン定数を使用する

テストでDBにデータを直接挿入する場合、文字列リテラルではなく必ずドメイン定数を参照する。

#### 修正前

```go
&db.SomeRecord{
    Provider: "A8NET",
    Status:   "PUBLIC",
    Category: "TEST",
}
```

#### 修正後

```go
&db.SomeRecord{
    Provider: domain.ProviderA8net.String(),
    Status:   domain.StatusPublished.String(),
    Category: domain.CategoryShopping.String(),
}
```

## 教訓

1. **機能分割によるテスト作成はデータ品質の盲点を生む**: テスト担当者は自分のテストの成否にしか関心がなく、挿入するデータがドメイン全体として正当かどうかは検証されにくい
2. **Spanner共有環境では全テストのデータが相互に可視**: あるテストが挿入した不正データが、別テストの全件取得APIに影響しうる
3. **DBへの直接挿入では必ずドメイン定数を使う**: ハードコード文字列はドメイン定数との不一致を招きやすい。型安全でない `string` 型カラムは特に注意が必要
4. **ハードコード値の不一致は書いた本人のテストでは検出されない**: 問題が表面化するのは、別のテストがそのデータを読み取ったときであり、発見が遅れる
5. **CIのテスト分割がFlaky testを隠蔽する**: CIでジョブが分割されている場合、ローカルの一括実行でしか再現しないFlaky testが存在しうる
